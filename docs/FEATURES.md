# Octopush ŌĆö Feature Map

> **This file is the single source of truth for everything Octopush can do and how it does it.**
>
> **Maintenance mandate (binding):** Any change that adds, removes, or meaningfully alters a user-facing feature **must** update this document in the same change. A PR that ships a feature without updating `docs/FEATURES.md` is incomplete. Keep it exhaustive ŌĆö down to the smallest context-menu item, toggle, keyboard shortcut, and empty state. When you add a new surface, add its features here; when you delete one, delete its entry. This rule is also recorded in `CLAUDE.md` / `AGENTS.md`.
>
> _Conventions:_ each feature is one bullet ŌĆö **Name** ŌĆö what it does (user-facing). _Support:_ the files / Tauri commands / Zustand stores / mechanism that implement it. _Entry:_ how the user reaches or triggers it. File paths are relative to the repo root; symbol names are durable, occasional line numbers are approximate hints only.

**Octopush** is "The IDE for Agentic Developers ŌĆö eight arms, zero wasted tokens." It is a Tauri 2 desktop app (macOS-first) with a React 19 + TypeScript frontend and a Rust backend. Workspaces are git worktrees. Its visual identity is **Atelier in Onyx & Brass** (see `docs/design-system.md`).

---

## Architecture at a glance

- **Frontend** (`src/`) ŌĆö React 19 + TypeScript, Tailwind v4 (`@theme` tokens in `src/styles.css`), state in **Zustand** stores (`src/stores/`), all IPC funneled through `src/lib/ipc.ts` (`invoke` from `@tauri-apps/api/core`). CodeMirror 6 editor, xterm.js terminals, React Flow pipeline builder, Recharts usage charts.
- **Backend** (`src-tauri/src/`) ŌĆö Rust. Tauri commands in `commands.rs` (registered in `lib.rs`'s invoke handler, ~190 commands), state in `state.rs`, SQLite data layer in `db.rs`, errors via `error.rs` (`Result<T, AppError>`). Domain modules: `chat_engine`, `orchestrator/*` (Direct mode), `git_ops`/`git_lock`/`github`, `providers/*`/`provider_router`, `token_engine`, `issue_tracker/*` (Jira), `mcp/*` + `mcp_setup`, `pty_*`/`talk_shell`, `theme`, `settings`, `perf`, `skills`, `context_guard`.
- **Two bundled sidecar binaries** (same crate, separate `[[bin]]`):
  - **`octopush-pty-server`** ŌĆö an out-of-process PTY daemon that owns all pseudo-terminals so shells survive app restarts/updates (`src-tauri/src/bin/octopush-pty-server/*`).
  - **`octopush-mcp`** ŌĆö a read-and-author MCP server that exposes Octopush's pipelines/roles/projects/workspaces/runs to external CLIs over stdio (`src-tauri/src/bin/octopush-mcp/*`).
- **Three primary modes** per workspace ŌĆö **Talk** (AI chat), **Run** (terminals) / **Direct** (multi-agent pipeline orchestration), **Review** (diff, editor, AI review). A right-hand **Companion** panel adapts to the mode; a left **Workspace Rail** lists projects ŌåÆ workspaces.
- **Data locations** ŌĆö SQLite at `~/Library/Application Support/octopush/octopush.db`; settings/keys/themes/providers under `~/.octopush/` (`settings.json`, `providers.json`, `theme.json`); PTY daemon socket/pid/logs and scrollback under `~/.octopush/`. **Local-first, no telemetry** ŌĆö outbound traffic only to configured AI providers and to integrations the user enables (Jira, GitHub).

- **Cross-machine run History (Pro-real Part B / B1+B2)** ŌĆö A `History` button (top bar, right side) opens a global **History sheet** (`ModalShell`) listing the signed-in **Pro** user's terminal Direct runs **from every machine they've signed in on** ŌĆö task, status glyph/word, cost, machine name, and relative time ŌĆö read-only, with a **Refresh** control. - **Library sync (Pro)** ŌĆö A signed-in Pro user's **custom pipelines and roles follow them across machines**. Mutable templates ŌćÆ **per-item last-writer-wins** on an `updated_at` stamp (new column on `pipelines`/`roles`, bumped by every save; RFC3339 ŌĆö lexicographic = chronological), **enforced in SQL on the server** so racing machines never regress a newer edit. Deletions travel as **tombstones** (delete on machine A propagates; an edit made *after* the delete revives the item ŌĆö newest intent wins; a role still in use locally survives a tombstone). **Push:** fire-and-forget after every `save_pipeline`/`delete_pipeline`/`save_role`/`delete_role` + a whole-library heal-push on launch. **Pull:** on launch (once entitled) ŌĆö **roles first** (pipelines validate against them), pipelines re-validated on apply (invalid/unparseable items are skipped, never written), then the pipeline/roles stores reload. Builtins **never travel** in either direction (a pulled item claiming a builtin id/key is refused). Server: `/api/sync-library` (Turso `synced_library_items`, byte cap 64KB/item ŌĆö an oversized item is *skipped*, never vetoes the batch; quota = 1000 **net** rows/owner counting inserts only ŌĆö edits always land; tombstones pruned after 90 days; stamps validated: unparseable or >24h-future are skipped). Client pushes are **chunked** (Ōēż200/POST) so a large library's heal never 413s wholesale. A role tombstone refused because the role is in use locally **bumps the local stamp**, so the next push genuinely revives the item cloud-side. **Known limitation:** authoring or deleting via `octopush-mcp` (a pipeline or, now, a role) bypasses the push hook (the MCP process can't push) ŌĆö an MCP-deleted item resurrects on the next pull, and an MCP-authored one won't propagate; touch it once in-app to make it stick. Gated by `library.sync` (Pro; `UpgradeRequired` on pull). _Support:_ `src-tauri/src/sync.rs` (`SyncPipeline`, `SyncLibraryItem`, tombstones), `db` (`list_custom_*_for_sync`, `upsert_*_from_sync`, `pipeline_stage_drafts`, LWW stamps), commands `library_sync_push_all`/`library_sync_pull` + mutation hooks; `App` launch effect. _Entry:_ automatic (Pro).

### Toasts & notifications
- **Crew notifications (native)** ŌĆö A macOS notification when a crew **needs a decision** or **finishes** ŌĆö the first piece of "crews that work while you don't". Triggers map to the orchestrator's REAL signals: **needs-you ŌåÉ `run://checkpoint` carrying `reason: "decision"`** (gate, halted stage, loop-at-cap, budget park; the director's own "pause at the next stage" emits `reason: "director"` and stays **silent** ŌĆö the user's own action is never a ping; a `paused` stage-update is NOT the signal: gates/halts never emit one); **finished ŌåÉ stage-update transition to `completed`** (with the cost; `aborted` = the director's own hand). **Asymmetric focus policy:** needs-you fires **always** (`document.hasFocus()` stays true on a locked screen ŌĆö a silent gate costs hours, a banner while watching costs nothing); finished stays quiet while focused (informative, not actionable). **No per-stage dedupe by design** ŌĆö the orchestrator never re-emits for a still-parked stage, and a loop-back re-parking the same stage row genuinely needs a fresh ping. Bounded session maps; surrogate-safe body trims; `sendNotification` called directly (desktop permission is delivery-time on macOS ŌĆö no stale grant cache). Toggle: Settings ŌåÆ General (default ON, persisted `octo-notify-prefs`). _Support:_ `src/lib/crewNotifications.ts` (pure `decideCheckpointNotification`/`decideCompletionNotification` + two listeners), `stores/notifyPrefsStore.ts`, `workspaceStore.findWorkspaceName` (shared with Mission Control), `tauri-plugin-notification` (+ capability). _Entry:_ automatic.
- **Detached runs (crews survive quitting) ŌĆö Pro** ŌĆö Every Direct run a Pro user starts, resumes (checkpoint approve/reject/resume), or re-runs executes in a **detached segment worker** (`octopush-run-worker`, a sidecar like the PTY daemon): the app reserves a cross-process **worker lease** on the `runs` row (`worker_nonce`/`worker_pid`/`heartbeat_at`), spawns the worker in its own session (`setsid`), and the worker drives ONE segment (start ŌåÆ next pause/completion) with the same `Orchestrator` ŌĆö so **quitting Octopush no longer kills in-flight crews**; the `claude` CLI child lives in the worker's session and survives too. The **detached bridge** (app-side ~1.2s tick) re-emits the worker's persisted progress as the same `run://stage-update`/`run://cost`/`run://log`/`run://checkpoint` events, so Mission Control, the live journal, and crew notifications work unchanged; it **adopts** leases from a previous app session (replaying the running stage's current attempt into the live pane, capped at 200 rows) and **repairs** runs whose worker died (stale heartbeat ŌåÆ the standard interrupted/Resume flow + a needs-you ping). Controls route cross-process: stop-stage/pause write `stop_requested`/`pause_requested` flags the worker polls (~1s); abort needs nothing new (the status write is already durable); the director's own pause keeps `reason: "director"` (no false ping). Startup recovery (`recover_interrupted_runs`) **skips fresh-leased runs** ŌĆö the one change that lets a live worker coexist with an app relaunch. If the segment ends while the app is closed, the worker posts a **native notification** via `osascript` ("Crew finished" / "The crew needs you"); aborts stay silent. The worker never migrates the DB (`Db::open_without_migrations`), never reads the keychain, never syncs history (`Orchestrator::new_headless`; the bridge syncs on its behalf). Any spawn failure falls back to the in-process drive ŌĆö detachment never costs a run. Free stays fully in-process ŌĆö but is **no longer silent about it**: the Direct launcher's unattended-readiness affordance surfaces a locked `Unattended ┬Ę Pro` chip ŌåÆ `runs.detached` upsell (see "Unattended readiness"), so detachment is a discoverable Pro capability rather than an invisible fallback. Mission Control cards show a quiet `Moon` icon on running detached runs ("Working detached ŌĆö this crew keeps going even if you quit Octopush"). _Support:_ `src-tauri/src/bin/octopush-run-worker/main.rs`, `orchestrator/worker.rs` (spawn + lease reserve), `orchestrator/bridge.rs` (watch/replay/repair), `db.rs` lease + control-flag helpers + `repair_interrupted_run`/`reconcile_stale_leases`, `entitlement::feature::RUNS_DETACHED`, routing in `commands::{start_run, resolve_checkpoint, rerun_from_stage, stop_stage, request_run_pause}`; design record `docs/superpowers/plans/2026-07-12-detached-runs-phase1-design.md`. _Entry:_ automatic for Pro.
- **Routines (scheduled crews) ŌĆö Pro** ŌĆö A saved pipeline that fires on a **schedule** and drives itself, closing the "crews that work while you don't" pillar: set it once and a review runs every morning or a dependency sweep every few hours, each fire executing as a **detached** run (so it survives the app quitting once started). An in-app **scheduler** (`routines::spawn_routine_scheduler`, a 30s tokio tick beside the detached bridge) selects enabled routines whose `next_due_at` has passed and fires each through the **exact same guarded launch** a user's Begin uses ŌĆö the guard sequence (same-workspace concurrency, worker lease, monthly quota, parallel-runs gate, budget, first-run marker, detached-spawn + in-process fallback) was extracted into one shared `orchestrator::launch::launch_run` so `commands::start_run` and the scheduler can never drift. Phase 1 fires only while the app runs; a window missed while closed **catches up once** on the next tick (`next_due` jumps to the next future slot ŌĆö never N times). **Schedule kinds:** `interval` (every N minutes/hours) and `daily` (at HH:MM machine-local); `next_due` is a pure, unit-tested computation stored in **UTC** so the SQL due-comparison is valid across DST. **Workspace modes:** `fixed` (run in a chosen workspace; a fire is skipped while a previous run is active ŌĆö `workspace_has_active_run`) and `fresh` (a new worktree on a unique timestamped branch each fire ŌĆö the isolated "ship a change every day" mode; a runaway guard skips while the routine's own previous run is still active, and phase 1 requires `fresh` to be **daily** ŌĆö `validate_routine` ŌĆö because there's no auto-reaper yet, so one worktree/day is the bound; the git checkout runs on `spawn_blocking`). The scheduler advances `next_due` BEFORE any side effect so a crash can't double-fire; a run refused at launch is deleted (not aborted, which would pollute the monthly meter). The scheduler re-checks the `routines.scheduled` entitlement before every fire (a downgrade silently stops firing) and never kills its loop on a per-routine failure. The double-fire guard is the `next_due` advance written **before** any side effect; `last_fired_at`/`last_run_id` are stamped after the run is created. A fixed routine whose target workspace was deleted auto-disables (rather than skipping every window forever). **Optional pre-fire condition (`fire_condition`):** a routine may carry an optional shell command evaluated before each fire ŌĆö **exit 0 ŌćÆ fire, non-zero ŌćÆ skip** the window with **no run created and zero tokens** (and for `fresh` mode, **no throwaway worktree** ŌĆö the gate runs BEFORE the workspace is resolved/created, right after the busy check and after the crash-safe `next_due` advance). It's fully **generic** (any command, any pipeline): e.g. a routine that addresses new PR-review comments hourly skips the idle windows with `gh pr view --json reviewThreads -q 'ŌĆ”' | grep -q .`. The command runs via a login shell (`bash -lc`) so the user's PATH + `gh` auth are present ŌĆö the SAME trust model as a run's `run_command`/setup scripts (their command, their machine) ŌĆö with **cwd = the routine's workspace** (fixed mode ŌåÆ the fixed workspace's `worktree_path`; fresh mode ŌåÆ the project root, since the fresh worktree doesn't exist yet) and a **30s timeout** (`tokio::time::timeout`, the child killed on timeout) so a hung check can never stall the 30s scheduler tick. A spawn failure or timeout is a **fail-safe skip** (never a blind fire), surfaced so it's discoverable. **Legibility (`last_checked_at`/`last_outcome`):** every evaluation ŌĆö dispatch or skip ŌĆö stamps the outcome ("dispatched" / "condition not met" / "condition error: ŌĆ”" / "workspace busy" / "workspace unavailable") and a timestamp, so a routine that keeps skipping stays legible in the list ("condition not met ┬Ę 2m ago") instead of looking dead. **UI:** a **Settings ŌåÆ Routines** pane (new `Automation` group) ŌĆö declarative, zero new top-level chrome (Atelier-compliant): a routine list (name ┬Ę schedule ┬Ę project ┬Ę workspace mode ┬Ę `conditional` marker ┬Ę next-fire ┬Ę last-outcome/checked-ago ┬Ę pause toggle ┬Ę run-now ┬Ę edit ┬Ę delete) + a `ModalShell` editor (project ŌåÆ pipeline ŌåÆ schedule ŌåÆ workspace mode ŌåÆ brief ŌåÆ optional **"Fire only ifŌĆ”"** mono command field with an exit-0 explainer ŌåÆ optional budget; the project/pipeline/workspace pickers are keyboard-navigable Atelier `Listbox`es, not native `<select>`s). A non-Pro user sees an upgrade banner but can still see and delete/pause routines they authored while Pro. `run_routine_now` fires immediately through the **identical** fire path ŌĆö the condition gate included ŌĆö and reports honestly whether a run was dispatched or the window was skipped **and why** (`{ outcome, reason? }`), so "Run now" is an honest test of exactly what a scheduled fire would do; to force a fire regardless, the user clears the condition. Commands: `list_routines`/`create_routine`/`update_routine`/`delete_routine` (deletion ungated) `/set_routine_enabled`/`run_routine_now` (all but delete Pro-gated via `require_feature_gate`). _Support:_ `src-tauri/src/routines.rs` (scheduler + pure `next_due`/`validate_schedule` + `evaluate_condition`/`FireOutcome`/`SkipReason`), `orchestrator/launch.rs` (shared guarded launch), `db.rs` `routines` table (`fire_condition`/`last_checked_at`/`last_outcome` columns) + CRUD (`insert_routine`/`list_due_routines`/`mark_routine_fired`/`set_routine_fire_result`/`workspace_has_active_run`), `entitlement::feature::ROUTINES_SCHEDULED`, `commands::{list,create,update,delete,set_enabled,run_now}_routine`; frontend `src/components/settings/RoutinesPane.tsx`, `lib/routineForm.ts` (pure form logic), `stores/routinesStore.ts`, `lib/settingsTabs.ts` (`Automation` group); design records `docs/superpowers/plans/2026-07-13-routines-phase1-design.md` + `docs/superpowers/plans/2026-07-15-routine-fire-condition-design.md`. _Entry:_ Settings ŌåÆ Routines.

## 2. Projects, Missions & Workspaces

Missions are the **first-level unit of intent** (build/fix/review/probe/design/perf/ops). A *code* mission (build/fix) owns exactly one **workspace**, and workspaces **are git worktrees** living as siblings of the project root under `<parent>/.octopus-worktrees/<branch>/`. The **main workspace** is special: its `worktree_path` equals the project root (git forbids the default branch checked out twice), so the project root *is* the main worktree. Isolation is a property a mission chooses along two axes ŌĆö git-state (`worktree`/`readonly`/`ephemeral`/`pr`) and execution (`none`/`sandbox`/`container`/`cloud`).

### Missions (threads of intent)
- **Missions as the first-level unit** ŌĆö Every workspace is born paired with a `build` mission; the pairing is idempotent, so a reused/adopted workspace keeps its mission and no workspace ever exists without one. (Design/probe missions arriving with later movements may have no worktree.) The active mission's intent surfaces as a glyph + label eyebrow above the workspace name in the ContextHeader. A partial unique index guarantees the core invariant: **never two active missions writing one checkout** ŌĆö readonly missions may share a checkout freely, but their agent context is always their own. _Support:_ `missions` table + nullable `mission_id` back-refs on `runs`/`chat_threads`/`terminals` (`src-tauri/src/db.rs`); the shared `src-tauri/src/mission.rs` (`create` / `ensure_for_workspace`, one code path also used by `octopush-mcp`); commands `create_mission` / `list_missions` / `get_mission` / `update_mission` / `archive_mission`; `src/stores/missionsStore.ts`; the `ContextHeader` intent chip (`INTENT_ICON`). A one-shot, `app_meta`-gated backfill re-parents every existing workspace into a `build` mission on upgrade. _Entry:_ automatic on workspace creation; intent chip in the header.
- **Sandboxed execution (macOS seatbelt) ŌĆö free** ŌĆö A mission's second isolation axis, `exec_isolation`, becomes real: with `exec_isolation = "sandbox"`, the mission's agents run **write-confined to the workspace**. The guarantee: an agent **cannot modify files outside its own mission** (other worktrees, `~/.ssh`, `~/.aws`, `~/.config`, `~/Documents`, system paths all stay read-only); **reads and network stay open** ŌĆö agents need both. Confinement covers **every execution surface**: (1) the **CLI substrate** (`sandbox-exec -f <profile> claude ŌĆ”`) ŌĆö enforced at the single spawn in `orchestrator/cli_runner.rs`, so the in-process orchestrator **and** the detached/unattended worker are covered identically (both use `CliRunner`; `exec_isolation` re-derived per stage, no cross-process state); (2) **in-process tool execution** ŌĆö TALK's inline tools and API-substrate DIRECT stages call `chat_engine::execute_tool`, which for a sandboxed mission runs BOTH `run_command` and `write_file` through the **kernel-enforced** seatbelt: `run_command` wraps its shell, and `write_file` writes via a seatbelt-wrapped `mkdir -p ŌĆ” && cat > "$1"` (target passed as a positional arg, content piped on stdin) so the **kernel resolves symlinks** ŌĆö a lexical check can't see through a committed `link -> $HOME`, so the write is done under the sandbox, not trusted to a string check (`sandbox::is_write_allowed` remains only a cheap pre-filter). A sandboxed TALK turn routes `run_command` away from the shared shell to the per-command sandboxed path. **No silent fallback:** if the sandbox can't be set up (non-macOS, missing driver, unwritable profile) ŌĆö or the mission asks for a not-yet-implemented tier (`container`/`cloud`) ŌĆö the work **fails with a readable error** rather than running unconfined; security never degrades quietly. The profile is `(allow default)` ŌåÆ `(deny file-write*)` ŌåÆ re-allow writes under the workspace + `$TMPDIR`/`/private/tmp` + `~/.claude` (+ the `~/.claude.json` state file via `(literal ŌĆ”)`), written to a temp file and cleaned up on exit. The active mission's sandbox state shows as a mute **`Shield`** glyph in the ContextHeader intent eyebrow. **Read-only missions** (review/probe, `git_isolation = "readonly"`) reuse the exact same machinery to become **read-only-by-construction**: `sandbox::sandbox_write_roots` folds both axes into one decision ŌĆö a readonly mission forces the sandbox ON with **empty** write roots (temp-only), so its agents read the checkout but can write nothing to it, regardless of the mission's own `exec_isolation`. Enforced at both surfaces (the run-stage resolution in `run_stage_once` and the TALK tool resolution in `send_agentic` both call `sandbox_write_roots`). **Out of scope (documented):** reads are intentionally open; MCP tool calls run in the user's own configured MCP server processes and aren't wrapped. **Gating: FREE** ŌĆö a free tier safer than the competition is the point; a product that charges for security reads as extortion. _Support:_ `src-tauri/src/orchestrator/sandbox.rs` (profile builder + argv wrapper + `prepare` + `is_write_allowed`), the fail-closed match in `orchestrator/cli_runner.rs`, the sandbox-aware `chat_engine::execute_tool` + `sandboxed_write` (threaded from `send_agentic` via `active_mission_for_workspace` and from `run_agentic_loop`/`ApiRunner` via `StageContext`), `exec_isolation` + `allowed_write_roots` on `StageContext`, `db::update_mission_exec_isolation`, `mission::validate_exec`; frontend `MissionCreator` Execution `Listbox`, `ContextHeader` `Shield` (`missionExecIsolation`). _Entry:_ wizard Step 2 ŌåÆ Execution ŌåÆ *Local sandbox*; Shield glyph in the header. (Container/cloud tiers arrive with later movements.)
- **Mission Logbook (worked time ┬Ę cost ┬Ę savings)** ŌĆö Every mission accrues a ledger of what it actually *cost* to run: **worked hours** (real activity time, not wall-clock), **spend**, and **savings vs an all-premium baseline**. Two data spines feed it: (1) **spend attribution** ŌĆö `spend_events` carries a nullable `mission_id`, stamped at write time from the active mission of the row's workspace (TALK/DIRECT/REVIEW; RUN-legacy and ad-hoc have no workspace so stay unattributed), with a one-shot backfill (`spend_events_mission_backfill_v1`); (2) **activity spans** ŌĆö a `work_spans` table records *when* a mission was actively worked, written by `record_activity(workspace, surface, source)` which **coalesces** repeated pings into one open span while work stays within a 10-minute idle window (so worked time is span-union seconds, never double-counted across overlapping surfaces). Hooks fire on real work across all four surfaces: a completed **TALK** turn (`chat_engine`), each **DIRECT** stage spend (`record_direct_spend`), **terminal** PTY output (the `spawn_or_attach_terminal` output hook, throttled to Ōēż1 beat/min per terminal so a live session's span grows without per-chunk cost), and **review** AI calls (`ai_complete` on a workspace). `logbook_summary(scope, from, to)` rolls it up: per-mission worked-second union (merged intervals) + cost + `savings = ╬Ż max(baselineŌłÆcost, 0)` + run/message counts + a per-surface breakdown. **Monetization:** the **per-mission** scope is **free** (you always see what your own mission cost); the cross-mission **project/global** scopes are **Pro** (`entitlement::feature::LOGBOOK_REPORTS`, gated by `require_logbook_reports`) ŌĆö the aggregate "where did my month go" view is the premium surface. - **Logbook Room (cross-mission rollup ŌĆö Pro) ŌĆö `ŌīśŌć¦L`** ŌĆö A full-screen `OverlayRoom` (Mission Control precedent) that aggregates the Logbook across missions: a **scope switch** (the current project ┬Ę all missions) ├Ś a **period switch** (7 days ┬Ę 30 days ┬Ę all time), a totals ledger in the header (worked time sage ┬Ę **saved** verdigris, savings-first ┬Ę spent brass), and a per-mission table (intent glyph + title ┬Ę worked ┬Ę spent ┬Ę saved ┬Ę runs ┬Ę messages, sorted by spend, `octo-rise-in` stagger). **Monetization boundary:** per-mission stays free (the Companion card); the cross-mission Room is **Pro** (`logbook.reports`). ### Notable implementation details
- **Worktree mechanics** ŌĆö `create_worktree` returns the path it actually used and is collision-proof: the libgit2 worktree **slot name and the working-tree path are decoupled from the branch**. It refuses up front if the branch is already checked out elsewhere (git's one-checkout-per-branch rule ŌåÆ friendly error). Otherwise it picks a **free slot name** (reclaiming a genuinely stale `.git/worktrees/<branch>` slot in place ŌĆö pruned with `valid+locked+working_tree`, then force-removed ŌĆö but stepping aside to `<branch>-2`, `-3`ŌĆ” when that slot still backs another live worktree, e.g. one whose branch was switched: the old "directory exists" bug) and a **free path** (clearing stale leftovers, suffixing past a live worktree). It never deletes a live worktree or the project root. Creation is serialized process-wide so concurrent creates can't race. `delete_worktree` prunes the slot **by path** (slot names aren't tied to the branch anymore) and never deletes the branch.
- **DB schema ŌĆö projects:** `id, name, path (UNIQUE), created_at, last_opened`, + `jira_project_key, closed_at, pinned, sort_order, tint`. **workspaces:** `id, project_id (FK CASCADE), name, task, branch, worktree_path, setup_script, status, created_at, last_active`, + `glyph, tint, test_command, linked_issue_key, issue_link_dismissed, from_branch`.
- **Customization dual-persistence** ŌĆö Workspace glyph/tint ŌåÆ DB only. Project name/tint ŌåÆ both localStorage (`projectCustomizations`) and DB, with a one-time migration into the backend.
- **Tint palette** (`monogram.ts`) ŌĆö 7 named tints (brass, verdigris, rouge, indigo, lavender, smoke, bone), each `{accent, bg}`.
- **URL parsing parity** ŌĆö `git_url.rs` (Rust) and `parseGitUrl.ts` (TS) handle HTTPS/SCP/ssh/multi-level GitLab/self-hosted hosts identically; the TS mirror keeps clone-name auto-detect instant.

- **Prompt quality doctrine (crew-output-quality pass)** ŌĆö Every producing role carries an **output contract** (named sections the next stage relies on: plan ŌåÆ Goal/Files/Steps/Edge cases/Acceptance criteria; repro ŌåÆ Repro/Root cause/Fix direction) and a definition of done (implement: build/lint/test if available, no stubs/TODOs, no drive-by refactors). Action roles gained safety bars (PR: never commit secrets/junk; merge: never on red/pending checks, proceeds with a note when the repo has none; release: abort on dirty tree). Guarded by tests (`builtin_role_prompts_are_purpose_specific_not_clones`, `reviewer_prompts_carry_a_severity_rubric_and_anti_rubber_stamp_bar`). _Support:_ `roles.rs`.

### Stage substrates & execution
- **API substrate (in-app agentic loop)** ŌĆö `ApiRunner` runs a stage through `run_agentic_loop`: resolves provider, composes system prompt, builds the input dossier, runs a tool-use loop bounded by `max_iterations`, returns exact token usage + cost. _Support:_ `orchestrator/runner.rs`, `agentic.rs`.
- **Per-stage reasoning effort (the cost/quality lever)** ŌĆö Each stage carries an optional `effort` (`Effort` enum: `low┬Ęmedium┬Ęhigh┬Ęxhigh┬Ęmax`; `null`/absent = **off**, no thinking ŌĆö today's default). It's **editor-set** in the pipeline builder (the Stage inspector's Reasoning control) ŌĆö and equally **authorable over `octopush-mcp`** (`create_pipeline`/`update_pipeline`, documented in `describe_pipeline_schema`) ŌĆö persisted on `pipeline_stages`, copied to `run_stages` at `create_run`, and read into `StageSpec.effort` ŌåÆ `LlmRequest.effort`. It's the atomic knob the cost model rests on: a cheap model thinking deeply on synthesis stages, barely thinking on mechanical ones. Applied on the **API substrate only** ŌĆö a CLI stage's agent manages its own reasoning, so the control is disabled and no effort pill renders for CLI stages. **Model-capability mapping** (pure, unit-tested, Anthropic): effort-capable models (Opus 4.x, Sonnet 5/4.6, Fable 5, unknown ŌåÆ the safe majority) get `output_config.effort:"<level>"` + `thinking:{type:"adaptive"}`, where the **level is clamped per model** (`effective_effort_level`) so we never 400 ŌĆö only Opus 4.7/4.8, Sonnet 5, Fable 5, Mythos 5 accept `xhigh`; Opus 4.6 / Sonnet 4.6 (the default) fold `xhigh`ŌåÆ`high` but keep `max`; Opus 4.5 / unknown fold both `xhigh` and `max`ŌåÆ`high`. The thinking-but-not-effort models (`haiku-4-5`, `sonnet-4-5`, `sonnet-4-0`) instead get `thinking:{type:"enabled", budget_tokens:N}` (N = low 4096 / med 8192 / high 16384 / xhigh┬Ęmax 24576, clamped `[1024, max_tokens-1]`) with **no** `output_config` (they 400 on it). Classification is a strict **allowlist** (`is_effort_model` / `is_budget_model`): a legacy (`claude-3-5-*`) or unknown id matches neither, so it gets **no thinking params at all** (effort silently ignored ŌĆö never 400 a non-thinker). `temperature` is never sent (current models 400 on it with thinking). Effort also raises the `max_tokens` floor (`max_tokens_for`: none/low/med 32768 ┬Ę high 48000 ┬Ę xhigh/max 64000) so deep thinking doesn't truncate the answer. With thinking + tool use, a completed assistant turn's **full content-block array is captured verbatim and replayed in order** (`LlmResponse.raw_content` ŌåÆ `AssistantWithTools.raw`) so signed thinking survives and interleaved thinking/tool_use ordering is preserved (reordering 400s); the max_tokens-truncation retry path deliberately replays with **empty** raw (a cut-off turn can carry an unsigned partial thinking block) and rebuilds from text + tool_uses. OpenAI-compat maps `effort` ŌåÆ `reasoning_effort` (low/med/high; xhigh┬Ęmax fold to high) **only on reasoning models** (o1/o3/o4, GPT-5 ŌĆö omitted on gpt-4o), and picks the token-cap key by class: reasoning models use `max_completion_tokens` (they 400 on `max_tokens`), chat models use `max_tokens`; both pass through unclamped. _Support:_ `providers/mod.rs` (`Effort`, `raw_content`), `providers/anthropic.rs` (`is_effort_model`/`is_budget_model`/`effective_effort_level`/`thinking_json`), `providers/openai_compat.rs` (`openai_supports_reasoning`), `orchestrator/agentic.rs` (`max_tokens_for`), `db.rs` (`effort` column on `pipeline_stages`/`run_stages`), `builder/StageInspector.tsx`. Design record: `docs/superpowers/plans/2026-07-14-per-stage-effort-design.md`.
- **Automatic model escalation (per stage)** ŌĆö A stage can carry an optional **escalation policy** ŌĆö `escalate_model` (a stronger model to retry with) and/or `escalate_effort` (a higher effort to bump to on the retry, API-only like base effort). A stage **has a policy iff either field is set**. It's **editor-set** in the Stage inspector's **"Escalate on failure"** control (an escalate-model picker with a "ŌĆö none ŌĆö" clear + an escalate-effort segmented control, disabled for CLI with the same note as the base effort) ŌĆö and equally **authorable over `octopush-mcp`** (`create_pipeline`/`update_pipeline`, documented in `describe_pipeline_schema`) ŌĆö persisted on `pipeline_stages`, copied to `run_stages` at `create_run`. **Trigger = a stage FAILS** (`StageStatus::Failed` ŌĆö includes the agentic loop exhausting `max_iterations` unfinished): the drive's Failed arm calls `try_escalate`, which retries the stage **once** at the strong tier before halting ŌĆö "pay the expensive tier on a *proven* failure, not by default". `try_escalate` re-reads the fresh stage row (the Failed arm's clone is pre-run state) and mirrors the re-run reset the reject/loop-back paths use: it **archives** the failed base-tier attempt to `stage_iterations`, **retires** its spend (so `recompute_run_cost` keeps counting it and the budget gate stays honest), **resets** the stage to pending **preserving** the existing feedback (and loop counter/session), sets the sticky `escalated` flag, and appends a live journal notice (`Ōåæ Escalating to {model} after a failed attempt`); the drive `continue`s and re-runs the now-pending stage. It escalates only when the tier would ACTUALLY change (`escalate_model` ŌēĀ base, or an API-substrate `escalate_effort` ŌēĀ base) ŌĆö a CLI effort-only or same-model policy halts instead of burning an identical retry. **Spec resolution** (`run_stage_once`): an escalated stage resolves `agent_model` = `escalate_model` (falling back to the base when only effort was bumped) and `effort` = `escalate_effort.or(base)` ŌĆö the **base fields stay on the row** for history/display. **Bounded to exactly one escalation**: a second failure hits `try_escalate` ŌåÆ already-escalated ŌåÆ halt. Because the flag is sticky, any later re-run (a review loop-back, a manual reject) also uses the strong tier for free. **`escalate_model` applies to BOTH substrates** (a plain model swap on the spec; the CLI runner reads `agent_model` too); **`escalate_effort` is API-only**. **A blocked stage (`ask_director` ŌåÆ `awaiting_checkpoint`) and an aborted run NEVER escalate** ŌĆö only the Failed arm calls `try_escalate`. **Budget/detached-compatible for free**: the per-run budget gate runs before a stage starts (an escalated retry over budget pauses for budget), and escalation converts a would-be-halt into a `continue` within the same drive segment, so a detached worker re-runs the escalated stage in-segment exactly as the in-process app does. A run-stage that escalated shows a brass `Ōåæ {model}` badge on its `RunFlow` meta line; the authoring surfaces (`StageNode`, `StageFlow`) show the policy as a brass `Ōåæ {model}` hint. **Ungated** (a reliability property, like per-stage effort and the escape valve). _Support:_ `orchestrator/mod.rs` (`try_escalate`, `drive_inner` Failed arm, `run_stage_once` spec resolution), `db.rs` (`escalate_model`/`escalate_effort` on `pipeline_stages`+`run_stages`, `escalated` on `run_stages`, `set_run_stage_escalated`, `RunStageRow`/`PipelineStageRow`/`StageDraft`), `builder/StageInspector.tsx` + `builder/graph.ts`, `RunFlow.tsx`/`direct/StageFlow.tsx`/`builder/StageNode.tsx` badges, `lib/ipc.ts`. Design record: `docs/superpowers/plans/2026-07-14-auto-escalation-design.md`.
- **Agentic loop internals** ŌĆö Per-stage tool allowlist; transient-error retry with backoff + journal narration; proactive rate-limit throttle; tool results capped at 24KB fed back; cancel flag checked each iteration; an unfinished loop is a **failure** (usage preserved). _Support:_ `agentic.rs`.
- **CLI substrate (headless Claude Code)** ŌĆö `CliRunner` spawns `claude -p --output-format stream-json --verbose --model ŌĆ” --append-system-prompt ŌĆ” --permission-mode bypassPermissions --max-turns N` in the worktree, streams NDJSON, parses the terminal `result` event. _Support:_ `cli_runner.rs`.
- **CLI environment & PATH resolution** ŌĆö Captures the user's full login+interactive shell env once (`SHELL -lic 'env -0'`) so a GUI-launched app inherits PATH/exports; resolves the `claude` executable across login PATH Ōł¬ inherited Ōł¬ common dirs. _Support:_ `login_shell_env`, `resolved_cli_path`.
- **CLI timeouts** ŌĆö Idle (300s silence) + absolute (3600s) with distinct honest messages; stderr drained concurrently. _Support:_ `cli_runner.rs`.
- **Uniform runner abstraction** ŌĆö Both substrates implement `AgentRunner::run(stage, input, ctx) -> StageOutcome`; the orchestrator never branches on substrate. _Support:_ `runner.rs`.
- **Stage input dossier (cross-stage handoff)** ŌĆö Each stage receives the run task, a one-line pipeline breadcrumb, and the **freshest artifact of each kind** from feeding stages (transitive flow-ancestors; sibling branches don't leak), each capped at ~16K chars. **Reviewers see the actual code:** when feeding artifacts ref the worktree, the **live diff (staged + unstaged ŌĆö a CLI agent habitually `git add`s) is captured and embedded** in the dossier between `BEGIN/END GIT DIFF` markers (not a ``` fence ŌĆö a diff of a markdown file contains fence lines), capped head+tail **with an explicit truncation notice** and a shared-workspace caveat ŌĆö so `code_review`/`verify`/`test` judge the real changes, not the implementer's prose summary. Best-effort: capture failure/empty diff falls back to the "inspect with your tools" hint; skipped for CLI-resume stages (input discarded). _Support:_ `orchestrator::full_worktree_diff` (shared with the stage diff snapshot), `assemble_stage_input`, `StageInput.worktree_diff`, `user_input_for`.

- **Ship a GitHub issue** ŌĆö In the launcher's brief card: a quiet `Github` entry ("ship an issue") opens a picker of the repo's **open GitHub issues** (`gh issue list` in the login shell, mirroring `list_prs`). **Preflight first** (`github_ship_readiness`): no github.com origin ŌåÆ honest dead end; `gh` unauthenticated ŌåÆ "run `gh auth login`" guidance ŌĆö the crew's `pull_request` stage would otherwise fail mid-run AFTER building the change. ### Checkpoints, gates & review loops
- **Checkpoint gates** ŌĆö A `checkpoint=true` stage parks `awaiting_checkpoint` + run `paused` on completion. _Support:_ `checkpoint` column; `drive_inner`.
- **Review loops ŌĆö two modes** ŌĆö A review stage (`can_loop`) can route work back to a `loop_target`. **Gated:** the run pauses; the human chooses Send-back vs Approve-anyway. **Auto:** the orchestrator loops targetŌåöreview automatically on a parsed `VERDICT: CHANGES_REQUESTED` until it passes or hits the cap. _Support:_ `LoopMode`; loop columns; `parse_verdict`.
- **Loop-back mechanics** ŌĆö Resets the contiguous `[target..=review]` range (restricted to flow-ancestors) to `pending`, forwards the reviewer's findings as feedback, archives erased attempts, retires their cost, bumps the loop counter, re-drives. Cap reached ŌåÆ escalates to a checkpoint. _Support:_ `loop_back`, `db.increment_loop_iteration`, `db.retire_stage_cost`.
- **Checkpoint actions (full vocabulary)** ŌĆö `resolve_checkpoint`: **Approve** (accept & continue; a failed stage's partial work is accepted via a synthesized artifact), **Reject** {feedback, model_override, max_turns_override} (re-run the single stage), **SendBack** {feedback} (loop the review back to its target), **Resume** {max_turns_override} (recover a halted stage; CLI `--resume`s the same session), **Discard** (revert the worktree to the failed stage's baseline), **Edit** (artifact edited out-of-band; continue), **AnswerBlocker** {answers} (answer a stage that parked itself via `ask_director` ŌĆö see the escape valve below), **Abort**. _Support:_ `CheckpointAction`; `runsStore.resolve`; `RunControlBar`.
- **Escape valve ŌĆö ask the director (`ask_director`)** ŌĆö A DIRECT stage that hits a decision only the director can make (a genuine ambiguity, a missing spec/credential, contradictory requirements) **stops and asks instead of guessing**: it calls the `ask_director` tool with a one-line `summary` and structured `questions`, each carrying a **recommended default**. The tool is appended to the agentic loop **after** the per-stage allowlist filter (so it survives a review stage's read-only tools) and is **DIRECT-only** ŌĆö never in TALK's shared `build_llm_tools`. On the call the loop **stops** (asking supersedes acting: any co-requested tool_use is discarded; the `BlockedAsk`/`BlockedQuestion` structs are tolerant ŌĆö every field defaults and the underscore fields accept both camelCase and snake_case, so a well-formed multi-question ask never collapses; only genuinely garbage input degrades, and even then it salvages **all** array elements) and the runner returns a **blocked outcome**; the drive persists the questions to `run_stages.blocked_questions`, parks the stage `awaiting_checkpoint`, and fires the **same** `emit_checkpoint(reason:"decision")` a gate uses ŌĆö so the crew "needs you" notification and the attention beacon fire unchanged (no new `StageStatus`; a block is just another checkpoint, distinguished by `blocked_questions IS NOT NULL`). The director answers via **`answer_blocker(run_id, stage_id, answers)`**: the decisions (each paired with its recommended default) become the stage's re-run **feedback**, `blocked_questions` clears, the asking spend is retired, and the stage **re-runs with the answers injected** (reusing the rejectŌåÆfeedbackŌåÆre-run substrate ŌĆö a block resolves by re-running, not a mid-loop resume). A block is **answer-only**: `apply_checkpoint_action` refuses Approve/Reject/SendBack on a stage with `blocked_questions` (only AnswerBlocker, or Abort, proceed) and refuses AnswerBlocker on a non-block ŌĆö a stale click can't mark a block done with an empty hand-off. A block also stamps `started_at`, so it is **not** treated as an editable un-begun park (`update_run_stage`/`StageFocus` require `started_at IS NULL`, not just "no artifact"). **Recommended-default fast path:** the answer UI pre-fills each input with the default, so accepting is one click. **API-substrate only** (CLI stages own their own interaction model): the carve-out is a separate `ASK_DIRECTOR_CLAUSE` appended by `compose_system_prompt(can_ask_director=true)` for API stages **only** ŌĆö CLI stages keep the strict never-ask preamble intact (they have no such tool). **Ungated** (core trust, like crew notifications). _Support:_ `agentic.rs` (`ask_director_tool`/`parse_ask_director`, `AgenticResult.blocked`), `types.rs` (tolerant `BlockedAsk`/`BlockedQuestion`, `StageOutcome.blocked`, `CheckpointAction::AnswerBlocker`), `runner.rs`, `orchestrator/mod.rs` (`run_stage_once` block-park, `format_blocker_feedback`, `apply_checkpoint_action` answer-only guard), `db.rs` (`run_stages.blocked_questions` + `set_run_stage_blocked`; `update_run_stage` un-begun guard), `commands::answer_blocker`, `roles.rs` (`ASK_DIRECTOR_CLAUSE` + `compose_system_prompt(can_ask_director)`); frontend `RunControlBar` `BlockedAskBar`, `MissionControl` block indicator, `StageFocus` un-begun guard, `runsStore.answerBlocker`, `ipc.answerBlocker`/`BlockedAsk`; design record `docs/superpowers/plans/2026-07-14-escape-valve-design.md`. _Entry:_ automatic when a stage blocks; answered from the decision bar.
- **Decision bar UI (state-adaptive)** ŌĆö `RunControlBar.tsx` switches on status: TerminalBar ("Run it again"), **DraftBar** (a STAGED run ŌĆö e.g. authored by `octopush-mcp` `create_run` ŌĆö mono `draft` word, serif-phrase primary CTA **"Begin this run"** ŌåÆ `runsStore.start` ŌåÆ `start_run(runId, null)` + detail refresh, and a **discard** control via `abort_run`; a refused start (quota/concurrency gate) shows the upgrade sheet and leaves the draft intact ŌĆö never a dead end), DecisionBar (checkpoint/halt, with Resume/Re-run turn-budget Stepper, Accept-partial, Discard-with-confirm, "why this halted" disclosure, feedback editor for reject/send-back, loop-state banner), **BlockedAskBar** (the escape valve: when a parked stage carries `blockedQuestions`, the decision gives way to **"The crew asked you"** ŌĆö one input per question pre-filled with its recommended default, plus **"Send answers"** and **"Accept all defaults"**). While a run is simply running the bar renders **nothing** ŌĆö pause/stop/abort live as the header's `IconButton`s instead (see Director controls above), and the single brass beacon (`lib/beacon.ts` `beaconAnchor`) sits on the DraftBar/DecisionBar's primary CTA or, mid-run, on the running stage's card in `RunFlow` ŌĆö never on this bar. A started draft behaves exactly like a launcher-started run (checkpoints, gates, ledger); `hasExecutingRun` still counts only running/paused. _Entry:_ bottom command bar.

### Notable implementation details
- **Orchestrator architecture** ŌĆö `Orchestrator` (`orchestrator/mod.rs`) drives runs as background tokio tasks, one stage at a time, pausing at checkpoints; enforces a single active drive per run, holds per-run cancel flags + pause requests, routes every emit through a `PersistingSink`. The `EventSink` trait decouples it from Tauri for testability.
- **Drive loop** ŌĆö Pick lowest non-`done` stage; complete the run if none; check budget gate then director-pause; run the stage; on a block (`ask_director`) ŌåÆ paused+checkpoint carrying the questions; on Failed ŌåÆ **try one automatic escalation** (`try_escalate`: a stage with an escalation policy that hasn't used it retries once at the strong tier, else) paused+checkpoint; on Done ŌåÆ auto-loop verdict decision before the gated pause. Aborts mid-stage always win.
- **DB tables** ŌĆö `pipelines`, `pipeline_stages` (+ pos/parents/tools/custom_name/instructions/loop/max_iterations/effort/escalate_model/escalate_effort), `runs` (+ cost/baseline/reference_model/retired/budget/linked_issue_key), `run_stages` (private per-run copy + status/tokens/cost/artifact/feedback/error/loop/diff_snapshot/session_id/resume_pending/baseline_commit/effort/blocked_questions/escalate_model/escalate_effort/escalated), `roles`, `stage_log`, `stage_iterations`, `run_events`. The `effort` column (nullable TEXT, the `Effort` token; NULL = off) is additive on `pipeline_stages` and `run_stages`; `blocked_questions` (nullable TEXT, JSON `BlockedAsk`) is additive on `run_stages` (the `ask_director` escape valve); the escalation policy ŌĆö `escalate_model` (nullable TEXT), `escalate_effort` (nullable TEXT) ŌĆö is additive on both, and `escalated` (INTEGER DEFAULT 0, sticky run-state) is additive on `run_stages`.
- **Tauri events** ŌĆö `run://stage-update`, `run://cost`, `run://checkpoint`, `run://error`, `run://log`. Subscribed once in `runsStore`.
- **Worktree as the blackboard** ŌĆö Stages never commit (handoff = worktree + structured artifact); all runs in a workspace share the one worktree, which is why only one executes at a time. A completed run leaves changes uncommitted for Review mode.
- **Shared micro-track primitive** ŌĆö `direct/StageDots.tsx` is the universal stage-dot track: "status" tone colors by run state, "shape" tone is the neutral launcher-ticket variant (every dot sage, only the gate ring carries meaning). Now consumed by `PipelineTicket`'s ticket shape line.

### Notable implementation details
- **git2 vs shell-out boundary** ŌĆö Read/inspect ops use libgit2 (status, diffs, log, blame, branch/stash list, create_branch, tags, worktree lifecycle). - **Auth model** ŌĆö Push/pull/fetch/switch/commit/cherry-pick/continue/abort run via the user's **login shell** (`$SHELL -l -c`) so SSH agents, the osxkeychain helper, and `~/.gitconfig` behave as in a terminal. - **The git lock** ŌĆö `git_lock.rs` keeps a global per-path async mutex so two mutating ops on the same worktree can't interleave; read-only commands take no lock.
- **PR state classification** lives entirely in `github.rs`; the frontend only maps `PrState` ŌåÆ color/glyph. The pure classifiers (`classify_pull`, `classify_continue`, `friendly_switch_error`, `pr_from_json`, `parse_git_url`) are heavily unit-tested.

## 7. Terminals, PTY Daemon & Sessions

### PTY Daemon ŌĆö out-of-process architecture
- **Standalone PTY daemon (`octopush-pty-server`)** ŌĆö A separate Rust binary that owns ALL pseudo-terminals so shells survive Octopush restarts/auto-updates. _Support:_ `src-tauri/src/bin/octopush-pty-server/*`; bundled as a sibling binary. _Entry:_ auto-spawned on app launch by `pty_daemon::ensure_daemon_running()`.
- **Detached spawn** ŌĆö Launched with `setsid()` + null stdio so it escapes Octopush's process group and is adopted by launchd. _Support:_ `pty_daemon.rs::spawn_detached`.
- **Binary resolution** ŌĆö Sibling of `current_exe()`, then `$PATH`, then `target/debug/ŌĆ”`. _Support:_ `resolve_daemon_binary`.
- **Unix domain socket** ŌĆö Newline-delimited JSON-RPC at `~/.octopush/pty-server.sock` (chmod 0700). _Support:_ `pty_daemon.rs`, `main.rs`.
- **PID-file double-start protection** ŌĆö `~/.octopush/pty-server.pid`; a second instance verifies the PID is alive AND is an `octopush-pty` process before exiting. _Support:_ `main.rs::acquire_pid_file`.
- **Protocol version handshake / stale-daemon replacement** ŌĆö Reuses a running daemon across version bumps if `protocol_version` matches (`EXPECTED_PROTOCOL_VERSION = 2`); on mismatch SIGTERMs the old daemon and spawns the new one. _Support:_ `query_daemon_protocol`, `kill_existing_daemon`.
- **Idle auto-exit** ŌĆö Exits when zero live PTYs AND zero clients for `OCTOPUSH_PTY_AUTO_EXIT_SECS` (default 3600s). _Support:_ `server.rs` idle thread.
- **Clean shutdown + crash recovery** ŌĆö SIGTERM/SIGINT removes pid/socket; a stale socket is unlinked on startup; orphan scrollback logs swept. _Support:_ `main.rs`, `storage.rs`.
- **fd-limit raising** ŌĆö Raises `RLIMIT_NOFILE` so terminal spawns don't fail with EMFILE. _Support:_ `main.rs::raise_fd_limit`.
- **Daemon log** ŌĆö `~/.octopush/pty-server.log`, rotated at 5 MiB. _Support:_ `storage.rs`.

### Daemon socket protocol
- **Request methods** ŌĆö `list_terminals`, `spawn`, `attach{since_seq?}`, `detach`, `write`, `resize`, `kill{signal?}`, `remove`, `shutdown`, `version`. _Support:_ `protocol.rs`, `server.rs::dispatch`.
- **Streaming events** ŌĆö `data{id,seq,bytes}`, `exit{id,code?}`, `error{id,message}`, `attention{id}`. _Support:_ `protocol.rs::Event`.
- **Per-connection model** ŌĆö Each connection on its own thread; a writer thread drains an unbounded channel; on disconnect, only that connection's attachments detach. _Support:_ `server.rs::handle_connection`.

### Scrollback persistence & replay
- **Dual buffering: ring + disk** ŌĆö Each chunk gets a monotonic `seq`, stored in a 64 KiB ring buffer AND appended to a disk log. _Support:_ `session.rs::push_output`.
- **Disk-backed log** ŌĆö `~/.octopush/pty-state/<id>.log`, capped at 1 MiB by rotation. _Support:_ `storage.rs`.
- **Scrollback replay on attach** ŌĆö Atomic snapshot under the lock: full disk log replay if `since_seq` is older than the ring, else ring backlog; live data streams after with no overlap. _Support:_ `server.rs::cmd_attach`.
- **Sessions outlive their shells** ŌĆö A session is kept after its shell exits (for replay) but its fds are released eagerly. _Support:_ `server.rs`, `session.rs`.

### Spawning, attaching & terminal lifecycle (daemon side)
- **Spawn a shell** ŌĆö Opens a PTY via `portable_pty`, launches `$SHELL` (or `/bin/zsh`) as a login shell in `cwd` with a curated env allowlist + `TERM`/`COLORTERM`, then merges caller env. _Support:_ `server.rs::cmd_spawn`.
- **Duplicate-id / respawn-in-place** ŌĆö A spawn for a live id is refused; an EXITED id is replaced in place (log deleted, seqs restart). _Support:_ `cmd_spawn`.
- **Write / resize / kill / remove** ŌĆö base64 write; PtySize resize; SIGTERM/SIGKILL kill (session retained); `remove` SIGKILLs the process group + deletes the log. _Support:_ `cmd_write`/`cmd_resize`/`cmd_kill`/`cmd_remove`.

### Attention detection (daemon-side)
- **"Waiting for input" detection** ŌĆö A 1s thread fires `Event::Attention` on (A) command-finished (fg pgroup returns to the shell) or (B) TUI-parked (fg child <1% CPU for two ticks + Ōēź2s quiet). _Support:_ `server.rs` attention thread, `session.rs::check_attention`.
- **False-positive gating** ŌĆö Requires Ōēź200 bytes since last alert; a 3s post-attach grace; baselines wiped on attach. _Support:_ `session.rs`.

### App-side daemon client
- **`DaemonClient`** ŌĆö One `UnixStream`; a reader thread multiplexes responses (per-`reqid` waiters, 5s timeout) and per-terminal event subscribers. _Support:_ `pty_client.rs`.
- **Single-subscriber-per-terminal routing** ŌĆö A re-attach replaces the prior subscriber so each byte renders once. _Support:_ `pty_client.rs::attach`.
- **Self-healing reconnect** ŌĆö On socket EOF the next call re-runs `ensure_daemon_running()` and swaps in a fresh socket. _Support:_ `try_reconnect`.
- **Stub client (degraded mode)** ŌĆö If the daemon binary is absent, a pre-broken stub lets Octopush still launch and heals on first reconnect. _Support:_ `pty_client.rs::stub`.
- **`PtyManager` + Tauri event bridge** ŌĆö Forwards `TermEvent`s as `pty://data`/`pty://exit`/`pty://attention`/`pty://reattached`. _Support:_ `pty_manager.rs`.
- **`spawn_or_attach` + `OutputHook`** ŌĆö Reattaches a running id (replays scrollback) or spawns fresh; an optional `OutputHook` runs per chunk for token scanning. _Support:_ `pty_manager.rs`.

### Backend commands & session model
- **Session commands** ŌĆö `create_session` (UUID, ContextGuard env, token hook, mark Active), `list_sessions`, `write_to_session`/`write_text_to_session`, `resize_session`, `kill_session`, `delete_session`. _Support:_ `commands.rs`.
- **Terminal commands (Run mode)** ŌĆö `list_terminals`, `create_terminal`, `rename_terminal`, `delete_terminal` (DB-first, then daemon `remove`). _Support:_ `commands.rs`; `terminals` table.
- **PTY-bridge commands** ŌĆö `spawn_or_attach_terminal` ŌåÆ `{mode:"Spawned"|"Reattached", pid?}`; `list_pty_sessions` (swallows daemon errors ŌåÆ empty for graceful degradation). _Support:_ `commands.rs`.
- **Session record & statuses** ŌĆö `Session{id, name, color, icon, project_root, agent(provider/model/ŌĆ”), token_budget, tokens_used, status, context_files, tags, ŌĆ”}`; status `Active|Idle|Paused|Completed|Error`. _Support:_ `session.rs`.

### Session restore & env injection
- **`restore_active_sessions`** ŌĆö On `.setup()`, re-spawns a PTY for each `Active`/`Idle` session, re-applying ContextGuard env + the token hook; marks `Error` on failure. _Support:_ `lib.rs`.

### Notable implementation details
- **Two protocols, not N vendors** ŌĆö Adding any vendor/gateway means choosing `anthropic` or `openai-compatible`; no per-vendor code.
- **Two stores split** ŌĆö providers.json (catalog incl. pricing/context) vs settings.json (`providerKeys`, `providerBaseUrls`, `last_pricing_refresh`). Keys never leave the device except in requests to providers.
- **Token-event schema** ŌĆö one canonical `spend_events` ledger holds **all** surfaces (TALK/RUN/REVIEW/DIRECT) with explicit `surface` + denormalized project/workspace attribution + `source_id` + idempotency; the `all_spend` view reads it (projecting `session_id = COALESCE(workspace_id, source_id)`), and project-scope budgets/savings join `all_spend ŌåÆ workspaces`. The old `token_events` table is retired to read-only history (migrated in via a one-shot backfill).
- **Budget enforcement points** ŌĆö TALK is now gated on **both** sides: the client pre-send fast-path (workspace/global + one-shot override) **and** an authoritative backend `check_budget` gate in `send_chat_message` (also enforces project-scope budgets the client can't see, and catches direct-IPC bypass; the client threads an `overrideBudget` flag so a conscious per-turn override skips it, and a backend `BudgetExceeded` ŌĆö `isBudgetExceeded` ŌĆö surfaces the same cap message + override affordance). REVIEW/`ai_complete` gates the same way; DIRECT keeps its between-stage run-budget park; post-turn threshold toasts unchanged. (DIRECT consulting global/project + TALK per-iteration gating + backend override grants remain a follow-up.) Two budget systems coexist: dollar-scoped budgets (the live UX) and a legacy per-session token budget (the token "remaining" gauge).
- **PTY token scraping is best-effort and fragile by design** ŌĆö the authoritative path is API usage recorded by `chat_engine`/`ai_complete`.

### The `octopush-mcp` standalone server binary
- **What/transport/scope** ŌĆö A sidecar `[[bin]]` exposing a **read-and-author** slice of Octopush over JSON-RPC 2.0 stdio, reusing `octopush_lib::db` against the same SQLite store (WAL-safe while the app is open). **Never executes runs or spends tokens**; the one git mutation it performs is materialising a workspace's worktree (`create_workspace`). _Support:_ `bin/octopush-mcp/main.rs`; doc `docs/octopush-mcp.md`.
- **Protocol handshake** ŌĆö Negotiates version (latest `2025-06-18`; also `2025-03-26`, `2024-11-05`), advertises `tools`, returns `instructions`; tool errors in-band (`isError: true`). _Support:_ `protocol.rs`.
- **Exposed tools (25)** ŌĆö _reference:_ `describe_pipeline_schema` (now also documents the `customRoles` authoring contract). _read:_ `list_pipelines`, `get_pipeline`, `list_roles`, `list_projects`, `list_workspaces`, `get_workspace`, `list_missions`, `get_mission`, `list_runs`, `get_run`, `list_routines`. _author:_ `create_pipeline` (validated DAG), `update_pipeline` (forks a builtin / edits a custom), `delete_pipeline` (builtins protected), `save_role`/`delete_role` (see role authoring below), `create_routine`/`update_routine`/`delete_routine`/`set_routine_enabled` (see routine authoring below), `link_workspace_issue` (metadata-only), `create_mission` (authors a mission ŌĆö build/fix/design/probe/ŌĆ” ŌĆö via the shared `mission.rs`; code work usually comes free via `create_workspace`), `create_run` (stages a DIRECT run in **`draft`** ŌĆö not started), `create_workspace` (ensures a worktree-backed workspace via the shared `workspace::create`; an explicit branch is used **verbatim** (validated git ref), task-derived branches are slugified; adopts an already-checked-out branch rather than failing; returns `status: created|adopted|existed|restored`; **pairs the workspace with a mission** (default intent `build`, or pass `intent`) so MCP-created workspaces match the app's "no workspace without a mission" guarantee; appears in the rail on next focus/refresh). `create_pipeline`/`update_pipeline` author the **same per-stage knobs the app builder does**, including per-stage reasoning **`effort`** and the model-escalation policy (**`escalateModel`**/**`escalateEffort`**) ŌĆö see the authoring contract below. The DB handle is a `Mutex<Db>` so `create_workspace` locks per-op, never across its git checkout. _Support:_ `tools.rs`.
- **Role authoring over MCP (`save_role`/`delete_role`)** ŌĆö an agent can author the **custom roles a good pipeline depends on**, not just reference the built-ins. `save_role` creates/updates a custom role from just `label` + `promptBody` (every other field defaults to the app's new-role values: `artifactKind` note, `environment` worktree, `defaultTools` [read_file, list_files], `defaultSubstrate` api, `canLoop`/`defaultCheckpoint` false, `tokenEstIn`/`tokenEstOut` 4000/1000); the `key` (a stage's `role`) is derived from the label with the **exact** app algorithm (`deriveRoleKey`) when omitted, and always returned. Routes through the same `db.upsert_role`/`get_role`/`role_in_use`/`delete_role` the app command uses ŌĆö **built-in keys are protected** (a save that would overwrite one, or a delete of one, is refused), out-of-enum `artifactKind`/`environment` is a **clean tool error** (typed `RoleDef` deserialize) and an unknown `defaultTools` entry or `defaultSubstrate` (stored on `RoleDef` as a plain `Vec<String>`/`String`, not enums) is rejected by an explicit check matching the advertised enums, and a role **still used by any pipeline/run stage** cannot be deleted. Like `delete_pipeline`, an MCP save/delete does **not** push to the Pro cloud library from the MCP process (edit once in-app to sync). _Support:_ `tools.rs` (`save_role`/`delete_role`/`list_roles`/`derive_role_key`), `db::upsert_role`/`role_in_use`. Design record: `docs/superpowers/plans/2026-07-15-mcp-authoring-parity-design.md`.
- **Routine authoring over MCP (`create_routine`/`update_routine`/`delete_routine`/`set_routine_enabled`)** ŌĆö an agent can author **scheduled crews** (Pro routines), the full in-app Routines surface minus immediate firing. **Author-only, safe by default:** `create_routine` creates the routine **DISABLED** unless `enabled:true` is passed ŌĆö authoring never silently schedules token spend; the user enables it in-app or the agent enables it on explicit instruction (`set_routine_enabled`). There is deliberately **no `run_routine_now`** (firing a crew *now* is execution, which stays in the app, like launching a run). An enabled routine fires only while the app runs and the plan includes `routines.scheduled` (**re-checked at fire time**, so an MCP-authored routine can't fire for a non-entitled user). `create_routine`/`update_routine` share one payload deserialized into the typed `RoutineInput` and validated through the **same** `octopush_lib::routines::{validate_routine, validate_schedule, next_due}` the app command uses, plus **referential-integrity checks** the agent needs (project/pipeline exist; `fixed` mode requires a `fixedWorkspaceId` that exists and belongs to the project; `fresh`ŌćÆ`daily`) so a dangling routine is rejected up front, not left to fail at fire time. `scheduleKind` `interval` (seconds-as-string Ōēź 60) or `daily` (`HH:MM`); `stageModelOverrides` (`[position, model]` pairs) round-trip through the stored JSON string; the optional `fireCondition` (exit-0 pre-fire gate) is carried through. `update_routine` leaves `enabled` untouched; `delete_routine` is ungated (matches the app). _Support:_ `tools.rs` (`create_routine`/`update_routine`/`delete_routine`/`set_routine_enabled`/`list_routines`/`build_routine_input`), `db::insert_routine`/`update_routine`/`set_routine_enabled`, `routines::{validate_routine,validate_schedule,next_due}`. Design record: `docs/superpowers/plans/2026-07-15-mcp-authoring-parity-design.md`.
- **Pipeline authoring contract** ŌĆö Same `┬¦3.7` validator as the app's `save_pipeline` (roles, loops on review roles, acyclic parents, substrate, tools subset, maxIterations 1..100, instructions Ōēż8000). Per-stage inputs include the **cost/quality knobs**: `effort` (`low|medium|high|xhigh|max`; null/omit = off; API-only; takes effect only on reasoning-capable models ŌĆö current Claude families + budget-path Haiku 4.5/Sonnet 4.5, where higher levels auto-clamp to the model's max; on legacy/unknown/non-Claude ids it is **silently ignored**, not clamped) and the escalation policy `escalateModel` (retry once on stage failure **only if it raises the tier** above `agentModel`; api+cli) / `escalateEffort` (api-only; retries only when higher than the base effort). Both round-trip through the exact `StageDraft`ŌåÆ`save_pipeline` path the app uses; `save_pipeline` normalizes a whitespace-only `escalateModel` to none, and an out-of-enum `effort`/`escalateEffort` is a clean per-stage tool error (never a panic). `describe_pipeline_schema` documents these fields with their real semantics, the effort/escalation **rules**, a `runtimeBehaviors` section (the `ask_director` **escape valve** ŌĆö an **API-substrate** stage may pause to ask the director when blocked; a `cli` stage never self-blocks, use a checkpoint/escalation there; plus **auto-escalation** guidance), and an annotated example that sets `effort` + `escalateModel`. _Support:_ `tools.rs`.
- **Registration & namespacing** ŌĆö `claude mcp add octopush -- <binary>`; tools appear as `mcp__octopush__<tool>`. Roadmap (not yet): execution control-plane, TALK/REVIEW surfaces, issue-tracker reads. _Support:_ `docs/octopush-mcp.md`.

## Appendix D ŌĆö Processes & on-disk locations

- **Processes** ŌĆö the Tauri app (`octopush`), the PTY daemon (`octopush-pty-server`, out-of-process, launchd-adopted), and on demand the MCP server (`octopush-mcp`, stdio, spawned by a CLI like Claude Code). Direct CLI stages spawn `claude -p ŌĆ”`; Run terminals spawn the user's login shell.
- **`~/.octopush/`** ŌĆö `settings.json` (keys, base URLs, git credentials, issue-tracker config, editor command, last pricing refresh), `providers.json` (provider/model catalog incl. pricing), `theme.json`, `projects/` (default scaffold location), `pty-server.sock` / `pty-server.pid` / `pty-server.log`, `pty-state/<id>.log` (terminal scrollback), `history/<session>.hist`.
- **`~/Library/Application Support/octopush/`** ŌĆö `octopush.db` (SQLite: projects, workspaces, sessions, terminals, chats, pipelines, runs, roles, journals, token events, budgets, ŌĆ”).
- **`~/.claude/`** ŌĆö read for `mcp.json` (MCP servers) and `skills/*` (user skills); Claude Code's own config receives the `octopush` MCP registration.
- **Per-project** ŌĆö `.octopus-worktrees/<branch>/` (workspace worktrees), `.claude/skills/*` and `.claude/mcp.json` (project-scoped skills/servers), and the migration from legacy `~/.octopus-sh` / `octopus-sh` dirs runs once on launch.
- **Privacy posture** ŌĆö local-first, no analytics/telemetry; outbound traffic only to configured AI providers and to integrations the user enables (Jira over HTTPS, GitHub via `gh`/REST).
