# Octopush — Feature Map

> **This file is the single source of truth for everything Octopush can do and how it does it.**
>
> **Maintenance mandate (binding):** Any change that adds, removes, or meaningfully alters a user-facing feature **must** update this document in the same change. A PR that ships a feature without updating `docs/FEATURES.md` is incomplete. Keep it exhaustive — down to the smallest context-menu item, toggle, keyboard shortcut, and empty state. When you add a new surface, add its features here; when you delete one, delete its entry. This rule is also recorded in `CLAUDE.md` / `AGENTS.md`.
>
> _Conventions:_ each feature is one bullet — **Name** — what it does (user-facing). _Support:_ the files / Tauri commands / Zustand stores / mechanism that implement it. _Entry:_ how the user reaches or triggers it. File paths are relative to the repo root; symbol names are durable, occasional line numbers are approximate hints only.

**Octopush** is "The IDE for Agentic Developers — eight arms, zero wasted tokens." It is a Tauri 2 desktop app (macOS-first) with a React 19 + TypeScript frontend and a Rust backend. Workspaces are git worktrees. Its visual identity is **Atelier in Onyx & Brass** (see `docs/design-system.md`).

---

## Architecture at a glance

- **Frontend** (`src/`) — React 19 + TypeScript, Tailwind v4 (`@theme` tokens in `src/styles.css`), state in **Zustand** stores (`src/stores/`), all IPC funneled through `src/lib/ipc.ts` (`invoke` from `@tauri-apps/api/core`). CodeMirror 6 editor, xterm.js terminals, React Flow pipeline builder, Recharts usage charts.
- **Backend** (`src-tauri/src/`) — Rust. Tauri commands in `commands.rs` (registered in `lib.rs`'s invoke handler, ~190 commands), state in `state.rs`, SQLite data layer in `db.rs`, errors via `error.rs` (`Result<T, AppError>`). Domain modules: `chat_engine`, `orchestrator/*` (Direct mode), `git_ops`/`git_lock`/`github`, `providers/*`/`provider_router`, `token_engine`, `issue_tracker/*` (Jira), `mcp/*` + `mcp_setup`, `pty_*`/`talk_shell`, `theme`, `settings`, `perf`, `skills`, `context_guard`.
- **Two bundled sidecar binaries** (same crate, separate `[[bin]]`):
  - **`octopush-pty-server`** — an out-of-process PTY daemon that owns all pseudo-terminals so shells survive app restarts/updates (`src-tauri/src/bin/octopush-pty-server/*`).
  - **`octopush-mcp`** — a read-and-author MCP server that exposes Octopush's pipelines/roles/projects/workspaces/runs to external CLIs over stdio (`src-tauri/src/bin/octopush-mcp/*`).
- **Three primary modes** per workspace — **Talk** (AI chat), **Run** (terminals) / **Direct** (multi-agent pipeline orchestration), **Review** (diff, editor, AI review). A right-hand **Companion** panel adapts to the mode; a left **Workspace Rail** lists projects → workspaces.
- **Data locations** — SQLite at `~/Library/Application Support/octopush/octopush.db`; settings/keys/themes/providers under `~/.octopush/` (`settings.json`, `providers.json`, `theme.json`); PTY daemon socket/pid/logs and scrollback under `~/.octopush/`. **Local-first, no telemetry** — outbound traffic only to configured AI providers and to integrations the user enables (Jira, GitHub).

## 1. Application Shell, Modes & Navigation

### App shell layout & structure
- **Vertical app frame** — The shell stacks `AppTopBar` (28px chrome) → main row (Rail · main · Companion) → `PerfMonitorBar` (footer), all `h-screen w-screen bg-octo-bg`. _Support:_ `src/App.tsx`. _Entry:_ always present once a project is open.
- **Unified header band + content row** — `<main>` renders a full-width `ContextHeader` band, then a content row holding the canvas (left) and Companion (right) flush beneath it. _Support:_ `src/App.tsx`. _Entry:_ automatic when an active workspace exists.
- **Always-mounted canvas (PTY survival)** — The left canvas column is never gated on `activeWorkspace`; empty-project/creator layers overlay it instead, so running terminals across projects keep their PTYs and xterm scrollback. _Support:_ `src/App.tsx`; `allTerminalRefs` flattens every `(workspace, terminal)` pair so all `TerminalPane`s mount simultaneously. _Entry:_ structural.
- **macOS traffic-light reservation + window drag** — Top bar reserves 78px for overlaid traffic lights and is a `data-tauri-drag-region`, so the window drags from any empty area. _Support:_ `src/components/AppTopBar.tsx`. _Entry:_ the top chrome bar.

### Three-mode (four-mode) workspace system
- **Mode model** — Four modes exist: `talk | run | review | direct`; display order is `["run","talk","review","direct"]`. Labels: Talk/Run/Review/Direct. _Support:_ `src/lib/modes.ts` (`WorkspaceMode`, `MODES`, `MODE_LABELS`, `MODE_SHORTCUTS`). _Entry:_ ModeSwitcher / shortcuts.
- **Per-workspace mode memory** — Each workspace remembers its own active mode (`modePerWorkspace`), defaulting to `talk`. Switching workspaces restores that workspace's last mode. _Support:_ `src/App.tsx`. _Entry:_ automatic.
- **Talk mode** — Conversational agent chat for the active workspace (`ChatView`); can open files in the editor and hand a command to Run. _Support:_ `src/App.tsx`. _Entry:_ Talk tab / ⌘⇧1.
- **Run mode** — Terminal surface; all PTYs mounted, only the active workspace's active terminal is visible; shows a "Start a new terminal." empty state when the workspace has zero terminals. _Support:_ `src/App.tsx` `RunEmptyState`. _Entry:_ Run tab / ⌘⇧2.
- **Review mode** — Three-pane review: `ReviewSidebar` (Changes+Files navigator) · `ReviewCanvas` (Diff/Editor toggle, hosts `EditorTabs`+`EditorPane`). Re-fetches git status/diff after commit/push/hunk actions. _Support:_ `src/App.tsx`. _Entry:_ Review tab / ⌘⇧3.
- **Direct mode** — Autonomous pipeline/run canvas (`DirectCanvas`), keyed by workspace id, seeded from `workspace.task` and `linkedIssueKey`. _Support:_ `src/App.tsx`. _Entry:_ Direct tab / ⌘⇧D.
- **ModeSwitcher (expanded, text)** — Content-width text buttons with a gliding+resizing brass-ghost indicator measured live from button geometry (ResizeObserver + window resize), 280ms ease; centered in the Companion header. _Support:_ `src/components/ModeSwitcher.tsx`. _Entry:_ Companion top bar.
- **Mode-tab attention pulse** — A Run/Talk tab pulses brass (`animate-attention-pulse`) when that workspace has a matching attention flag and the user is in a different mode; tooltip switches to "… needs your attention". _Support:_ `ModeSwitcher.tsx`; `attentionStore`. _Entry:_ automatic on background completions.
- **Mode-overlay crossfade** — Each mode canvas is an absolutely-positioned, opacity/visibility/pointer-events-gated overlay (200ms) so switching modes never unmounts the others. _Support:_ `src/components/ModeOverlay.tsx`. _Entry:_ structural.
- **Cross-mode "Run in terminal" hand-off** — From a Talk tool card, take a shell command to Run mode: switches mode and copies the command to the clipboard (deliberately not auto-executed). _Support:_ `src/App.tsx` `handleRunInTerminal`; toast "Command copied — paste into the terminal to run". _Entry:_ Talk chat tool action.

### ContextHeader (top header band)
- **Active-ticket header** — When a workspace resolves a Jira ticket (manual link wins, else branch-detected gated on project key + tracker configured), shows a `◈` brass diamond, the parent/grandparent ticket chain (clickable, opens in Jira), the ticket key (type-tinted), status name, and the summary in serif. _Support:_ `src/components/ContextHeader.tsx`; `useActiveIssue`, `parentIssuesStore`. _Entry:_ automatic header.
- **Degraded workspace header** — With no ticket, shows a "WORKSPACE" brass eyebrow + workspace name (animated on change). _Support:_ `ContextHeader.tsx`. _Entry:_ automatic.
- **Branch + base + unstaged indicator** — Right side shows a verdigris dot, `↳ branch`, optional "from <fromBranch>" (base-branch tooltip), and "· N unstaged" when changes exist. _Support:_ `ContextHeader.tsx`. _Entry:_ automatic.
- **PR chip** — Per-state pull-request chip (open ●/draft ◐/merged ✓/closed ✕) with `PR · #<number>` and ↗; click opens the PR URL in the system browser. A hover-revealed copy button copies the PR URL to the clipboard (toast: "PR URL copied") without opening the browser. _Support:_ `ContextHeader.tsx` `PR_STATE_STYLE`; `openPrByWs` cache fed by `findPrForBranch` polled every 60s; copy via `copyToClipboard`. _Entry:_ click the chip; hover for the copy icon.

### AppTopBar
- **Live mascot (The Octo) + wordmark** — Centered in the top bar: a 20px `OctoMark` next to the `Octopush` wordmark (`.brand-wordmark`, Fraunces), whose body language mirrors global app state via `useMascotState` (`src/hooks/useMascotState.ts`), priority **blocked > working > idle** — any workspace flagged in `attentionStore` **or any Direct run `paused`** (checkpoint gate / budget park — waiting on the director, matching Mission Control's needs-you triage) → `blocked` (frozen, eyes half-mast: stillness is the signal); else any chat streaming (`chatStore.streamingByWs`) or Direct run `running` (`runsStore`) → `working` (paddling at double tempo, eyes scanning); else `idle` (slow float, staggered paddle, blinks). Tooltip/aria reports the exact state ("Octopush — 2 agents working"). Decorative-plus-informative; no click behavior. _Support:_ `src/components/AppTopBar.tsx`; `icons/OctoMark.tsx`; `hooks/useMascotState.ts`. _Entry:_ top bar, centered.
- **Fleet chip (runs tray)** — Global top-bar chip (left side), the entry to Mission Control. Attention-split priority chain (the same `needsYou` predicate Mission Control's Needs-you band uses): any run waiting on the director → brass `{n} needs you`, pulsing (`octo-stage-pulse`) — the chip's only pulse trigger, per Law 2's fleet scope; else any run running with nobody waiting → quiet hairline `{n} in flight` (sage, no pulse); else runs **settled this session** (undismissed) → verdigris `✓ {n} done` (Lucide `Check`, not a literal glyph) — the entry no longer vanishes the moment background work finishes. **No cost figure in this chip** — the combined live-cost figure lives only in the Mission Control header. Hidden only when the board is truly empty. Clicking opens the **Mission Control room** (which replaced the old A2 `MenuSurface` popover — one canonical surface for the fleet). Hydrated on launch via `list_active_runs`; kept live by `run://*` events. _Support:_ `src/components/RunsTray.tsx`; `lib/runStatus.ts` (`needsYou`); `runsStore` (`settledAt`). _Entry:_ top bar, left side.
- **Mission Control (the fleet cockpit)** — A full-screen **room** (`OverlayRoom`, Settings precedent) showing **every Direct run across every workspace** as a live crew card, in three triage bands with Law 1 ink grading: **Needs you** (`paused` — gate / halted stage / budget park; full ink, brass border, FIFO by `statusSince`), **In flight** (`running`; `opacity-75`), **Settled** (runs that reached `completed`/`aborted` **this session** — they linger until dismissed, so nothing that finishes in the background silently vanishes; session-local `runsStore.settledAt`; `opacity-45` rising to `0.85` on hover/focus). Each card: status glyph/word + a fixed-width **time-in-state** slot (`mm:ss` rolling to `Hh MMm`, tabular — not a "WAITING N MIN" eyebrow) · workspace (serif) · task · **`StageDots`** micro-track (replaces the retired roman-numeral track; one dot per stage, colored by status) · a **fixed-slot live line** (running → the stage's live activity ticker via `lib/liveLine.lastActivity`; gate → "{stage} holds the gate · loop n/m"; **`ask_director` block → a distinct `?` glyph + "{stage} asked you · waiting on your answer"** (not the gate wording — an answer form is waiting, not an approval); halt → first error line, amber `⟳ stalled` for transient; settled → savings epitaph) · ledger foot (spent brass, saved verdigris) with hover actions — **abort** (two-step confirm) or **dismiss** (settled). Law 2's fleet scope: only the **longest-waiting** needs-you card pulses (`octo-stage-pulse`, FIFO index 0); every other needs-you card keeps a static brass border. Card click **jumps** to the run's workspace + Direct surface (no checkpoint resolution on-card — judging a gate needs the artifact/diff, so deeper intents resolve to the jump). Header: band counts + a combined live-cost figure over **active** runs only (`saved $X · Y% under all-premium · spent $Z` — this figure is Mission-Control-only; the `RunsTray` chip dropped it) + quiet `Plus` **"Send out a crew"** (→ Direct mode) + `ESC · CLOSE`. A crew completing gets the one sanctioned ceremony: a single solid `.octo-sweep` line (no gradient) across its ledger foot as it settles. Empty state: *"The floor is quiet."* + the dispatch CTA. Free users see their single run; starting a 2nd hits the existing `runs.parallel` gate → upgrade sheet. _Support:_ `src/components/MissionControl.tsx`, `src/components/direct/StageDots.tsx`, `primitives/OverlayRoom.tsx` (shared with Settings), `runsStore` (`settledAt`/`statusSince`/`dismissSettled`/`clearSettled`), `lib/liveLine.ts` (ticker vocabulary shared with `RunFlow`); design record `docs/superpowers/plans/2026-07-09-mission-control-design.md`. _Entry:_ fleet chip click · `⌘⇧M`.
- **Cross-machine run History (Pro-real Part B / B1+B2)** — A `History` button (top bar, right side) opens a global **History sheet** (`ModalShell`) listing the signed-in **Pro** user's terminal Direct runs **from every machine they've signed in on** — task, status glyph/word, cost, machine name, and relative time — read-only, with a **Refresh** control. Rows carry the depth-of-field row optic (`opacity-45`, rising to `0.85` on hover/focus). For **Free / signed-out** users the button instead opens the **upgrade sheet** (`feature: "history.sync"`), an upsell. _Mechanism:_ when a run reaches a terminal state (`completed`/`aborted`) the orchestrator **fire-and-forgets** a push of that run's **metadata only** (task/status/cost/summed tokens/per-stage role+model+status/timestamps — never journals, diffs, artifacts) to the Octopush **sync API** (`octopush-api.vercel.app/api/sync-runs`, same Vercel project as billing), authenticated with the Clerk **access token** (refreshed if expired) and Pro-re-checked **server-side**; the server (Turso/libSQL) owns each row by the authenticated Clerk `sub` (never trusts the client) and upserts by run id (terminal runs are immutable → append-only, no conflicts). On launch (Pro only) the app **backfills** this machine's terminal runs in one capped batch, then **pulls** the full history into a **local read-only `synced_runs` mirror** (kept separate from the FK-constrained `runs` table — pulled runs have no local workspace). **Drill-in (B2):** clicking a row fetches that run's **full story lazily** — per-stage **journals** (LiveEntry-shaped, rendered as inert mono lines with `iconForTool`), **artifact texts**, and **diff snapshots** (`DiffViewer`) — pushed once at terminal time (journals capped 200 entries/48KB per stage newest-win, artifacts 16KB, diffs 96KB head+tail; server backstop 1.5MB/blob, 2k rows/owner in `synced_run_details`) and fetched on demand (`history_run_detail` → `/api/sync-run-detail`; session-cached incl. the "none" answer). The detail view is a `FadeSwap` list↔detail drill-in with `StageDots`, role icons, and honest states (loading / retryable fetch error / "synced before journals were kept"). Synced values render as **inert text** (never HTML). Gated end-to-end by `history.sync` (`AppError::UpgradeRequired` on the sync commands; no-op push for non-Pro; server 403). _Support:_ `src/components/HistorySheet.tsx`, `historyStore`, `AppTopBar` History button, `App` launch sync; `src-tauri/src/sync.rs` (`SyncRun`, `build_run_payload`, `push_runs`/`pull_runs`), `auth::current_access_token`, `orchestrator::sync_run_history` (3 terminal sites), `db` (`app_meta` + `machine_id`, `synced_runs` mirror, `list_terminal_runs`); commands `history_list`/`history_sync_pull`/`history_sync_push_all`. _Entry:_ top bar → History. See [`docs/premium/pro-real-part-b-history-sync.md`](premium/pro-real-part-b-history-sync.md).
- **Library sync (Pro)** — A signed-in Pro user's **custom pipelines and roles follow them across machines**. Mutable templates ⇒ **per-item last-writer-wins** on an `updated_at` stamp (new column on `pipelines`/`roles`, bumped by every save; RFC3339 — lexicographic = chronological), **enforced in SQL on the server** so racing machines never regress a newer edit. Deletions travel as **tombstones** (delete on machine A propagates; an edit made *after* the delete revives the item — newest intent wins; a role still in use locally survives a tombstone). **Push:** fire-and-forget after every `save_pipeline`/`delete_pipeline`/`save_role`/`delete_role` + a whole-library heal-push on launch. **Pull:** on launch (once entitled) — **roles first** (pipelines validate against them), pipelines re-validated on apply (invalid/unparseable items are skipped, never written), then the pipeline/roles stores reload. Builtins **never travel** in either direction (a pulled item claiming a builtin id/key is refused). Server: `/api/sync-library` (Turso `synced_library_items`, byte cap 64KB/item — an oversized item is *skipped*, never vetoes the batch; quota = 1000 **net** rows/owner counting inserts only — edits always land; tombstones pruned after 90 days; stamps validated: unparseable or >24h-future are skipped). Client pushes are **chunked** (≤200/POST) so a large library's heal never 413s wholesale. A role tombstone refused because the role is in use locally **bumps the local stamp**, so the next push genuinely revives the item cloud-side. **Known limitation:** authoring or deleting via `octopush-mcp` (a pipeline or, now, a role) bypasses the push hook (the MCP process can't push) — an MCP-deleted item resurrects on the next pull, and an MCP-authored one won't propagate; touch it once in-app to make it stick. Gated by `library.sync` (Pro; `UpgradeRequired` on pull). _Support:_ `src-tauri/src/sync.rs` (`SyncPipeline`, `SyncLibraryItem`, tombstones), `db` (`list_custom_*_for_sync`, `upsert_*_from_sync`, `pipeline_stage_drafts`, LWW stamps), commands `library_sync_push_all`/`library_sync_pull` + mutation hooks; `App` launch effect. _Entry:_ automatic (Pro).
- **Scratchpad toggle** — `NotebookPen` button toggles the scratchpad split. _Support:_ `AppTopBar.tsx`; `scratchpadStore.toggleOpen`. _Entry:_ top bar, right side.
- **Settings button** — `Settings` icon opens Settings on the General tab. _Support:_ `AppTopBar.tsx`. _Entry:_ top bar, right side.

### Toasts & notifications
- **Crew notifications (native)** — A macOS notification when a crew **needs a decision** or **finishes** — the first piece of "crews that work while you don't". Triggers map to the orchestrator's REAL signals: **needs-you ← `run://checkpoint` carrying `reason: "decision"`** (gate, halted stage, loop-at-cap, budget park; the director's own "pause at the next stage" emits `reason: "director"` and stays **silent** — the user's own action is never a ping; a `paused` stage-update is NOT the signal: gates/halts never emit one); **finished ← stage-update transition to `completed`** (with the cost; `aborted` = the director's own hand). **Asymmetric focus policy:** needs-you fires **always** (`document.hasFocus()` stays true on a locked screen — a silent gate costs hours, a banner while watching costs nothing); finished stays quiet while focused (informative, not actionable). **No per-stage dedupe by design** — the orchestrator never re-emits for a still-parked stage, and a loop-back re-parking the same stage row genuinely needs a fresh ping. Bounded session maps; surrogate-safe body trims; `sendNotification` called directly (desktop permission is delivery-time on macOS — no stale grant cache). Toggle: Settings → General (default ON, persisted `octo-notify-prefs`). _Support:_ `src/lib/crewNotifications.ts` (pure `decideCheckpointNotification`/`decideCompletionNotification` + two listeners), `stores/notifyPrefsStore.ts`, `workspaceStore.findWorkspaceName` (shared with Mission Control), `tauri-plugin-notification` (+ capability). _Entry:_ automatic.
- **Detached runs (crews survive quitting) — Pro** — Every Direct run a Pro user starts, resumes (checkpoint approve/reject/resume), or re-runs executes in a **detached segment worker** (`octopush-run-worker`, a sidecar like the PTY daemon): the app reserves a cross-process **worker lease** on the `runs` row (`worker_nonce`/`worker_pid`/`heartbeat_at`), spawns the worker in its own session (`setsid`), and the worker drives ONE segment (start → next pause/completion) with the same `Orchestrator` — so **quitting Octopush no longer kills in-flight crews**; the `claude` CLI child lives in the worker's session and survives too. The **detached bridge** (app-side ~1.2s tick) re-emits the worker's persisted progress as the same `run://stage-update`/`run://cost`/`run://log`/`run://checkpoint` events, so Mission Control, the live journal, and crew notifications work unchanged; it **adopts** leases from a previous app session (replaying the running stage's current attempt into the live pane, capped at 200 rows) and **repairs** runs whose worker died (stale heartbeat → the standard interrupted/Resume flow + a needs-you ping). Controls route cross-process: stop-stage/pause write `stop_requested`/`pause_requested` flags the worker polls (~1s); abort needs nothing new (the status write is already durable); the director's own pause keeps `reason: "director"` (no false ping). Startup recovery (`recover_interrupted_runs`) **skips fresh-leased runs** — the one change that lets a live worker coexist with an app relaunch. If the segment ends while the app is closed, the worker posts a **native notification** via `osascript` ("Crew finished" / "The crew needs you"); aborts stay silent. The worker never migrates the DB (`Db::open_without_migrations`), never reads the keychain, never syncs history (`Orchestrator::new_headless`; the bridge syncs on its behalf). Any spawn failure falls back to the in-process drive — detachment never costs a run. Free stays fully in-process. Mission Control cards show a quiet `Moon` icon on running detached runs ("Working detached — this crew keeps going even if you quit Octopush"). _Support:_ `src-tauri/src/bin/octopush-run-worker/main.rs`, `orchestrator/worker.rs` (spawn + lease reserve), `orchestrator/bridge.rs` (watch/replay/repair), `db.rs` lease + control-flag helpers + `repair_interrupted_run`/`reconcile_stale_leases`, `entitlement::feature::RUNS_DETACHED`, routing in `commands::{start_run, resolve_checkpoint, rerun_from_stage, stop_stage, request_run_pause}`; design record `docs/superpowers/plans/2026-07-12-detached-runs-phase1-design.md`. _Entry:_ automatic for Pro.
- **Routines (scheduled crews) — Pro** — A saved pipeline that fires on a **schedule** and drives itself, closing the "crews that work while you don't" pillar: set it once and a review runs every morning or a dependency sweep every few hours, each fire executing as a **detached** run (so it survives the app quitting once started). An in-app **scheduler** (`routines::spawn_routine_scheduler`, a 30s tokio tick beside the detached bridge) selects enabled routines whose `next_due_at` has passed and fires each through the **exact same guarded launch** a user's Begin uses — the guard sequence (same-workspace concurrency, worker lease, monthly quota, parallel-runs gate, budget, first-run marker, detached-spawn + in-process fallback) was extracted into one shared `orchestrator::launch::launch_run` so `commands::start_run` and the scheduler can never drift. Phase 1 fires only while the app runs; a window missed while closed **catches up once** on the next tick (`next_due` jumps to the next future slot — never N times). **Schedule kinds:** `interval` (every N minutes/hours) and `daily` (at HH:MM machine-local); `next_due` is a pure, unit-tested computation stored in **UTC** so the SQL due-comparison is valid across DST. **Workspace modes:** `fixed` (run in a chosen workspace; a fire is skipped while a previous run is active — `workspace_has_active_run`) and `fresh` (a new worktree on a unique timestamped branch each fire — the isolated "ship a change every day" mode; a runaway guard skips while the routine's own previous run is still active, and phase 1 requires `fresh` to be **daily** — `validate_routine` — because there's no auto-reaper yet, so one worktree/day is the bound; the git checkout runs on `spawn_blocking`). The scheduler advances `next_due` BEFORE any side effect so a crash can't double-fire; a run refused at launch is deleted (not aborted, which would pollute the monthly meter). The scheduler re-checks the `routines.scheduled` entitlement before every fire (a downgrade silently stops firing) and never kills its loop on a per-routine failure. The double-fire guard is the `next_due` advance written **before** any side effect; `last_fired_at`/`last_run_id` are stamped after the run is created. A fixed routine whose target workspace was deleted auto-disables (rather than skipping every window forever). **Optional pre-fire condition (`fire_condition`):** a routine may carry an optional shell command evaluated before each fire — **exit 0 ⇒ fire, non-zero ⇒ skip** the window with **no run created and zero tokens** (and for `fresh` mode, **no throwaway worktree** — the gate runs BEFORE the workspace is resolved/created, right after the busy check and after the crash-safe `next_due` advance). It's fully **generic** (any command, any pipeline): e.g. a routine that addresses new PR-review comments hourly skips the idle windows with `gh pr view --json reviewThreads -q '…' | grep -q .`. The command runs via a login shell (`bash -lc`) so the user's PATH + `gh` auth are present — the SAME trust model as a run's `run_command`/setup scripts (their command, their machine) — with **cwd = the routine's workspace** (fixed mode → the fixed workspace's `worktree_path`; fresh mode → the project root, since the fresh worktree doesn't exist yet) and a **30s timeout** (`tokio::time::timeout`, the child killed on timeout) so a hung check can never stall the 30s scheduler tick. A spawn failure or timeout is a **fail-safe skip** (never a blind fire), surfaced so it's discoverable. **Legibility (`last_checked_at`/`last_outcome`):** every evaluation — dispatch or skip — stamps the outcome ("dispatched" / "condition not met" / "condition error: …" / "workspace busy" / "workspace unavailable") and a timestamp, so a routine that keeps skipping stays legible in the list ("condition not met · 2m ago") instead of looking dead. **UI:** a **Settings → Routines** pane (new `Automation` group) — declarative, zero new top-level chrome (Atelier-compliant): a routine list (name · schedule · project · workspace mode · `conditional` marker · next-fire · last-outcome/checked-ago · pause toggle · run-now · edit · delete) + a `ModalShell` editor (project → pipeline → schedule → workspace mode → brief → optional **"Fire only if…"** mono command field with an exit-0 explainer → optional budget; the project/pipeline/workspace pickers are keyboard-navigable Atelier `Listbox`es, not native `<select>`s). A non-Pro user sees an upgrade banner but can still see and delete/pause routines they authored while Pro. `run_routine_now` fires immediately through the **identical** fire path — the condition gate included — and reports honestly whether a run was dispatched or the window was skipped **and why** (`{ outcome, reason? }`), so "Run now" is an honest test of exactly what a scheduled fire would do; to force a fire regardless, the user clears the condition. Commands: `list_routines`/`create_routine`/`update_routine`/`delete_routine` (deletion ungated) `/set_routine_enabled`/`run_routine_now` (all but delete Pro-gated via `require_feature_gate`). _Support:_ `src-tauri/src/routines.rs` (scheduler + pure `next_due`/`validate_schedule` + `evaluate_condition`/`FireOutcome`/`SkipReason`), `orchestrator/launch.rs` (shared guarded launch), `db.rs` `routines` table (`fire_condition`/`last_checked_at`/`last_outcome` columns) + CRUD (`insert_routine`/`list_due_routines`/`mark_routine_fired`/`set_routine_fire_result`/`workspace_has_active_run`), `entitlement::feature::ROUTINES_SCHEDULED`, `commands::{list,create,update,delete,set_enabled,run_now}_routine`; frontend `src/components/settings/RoutinesPane.tsx`, `lib/routineForm.ts` (pure form logic), `stores/routinesStore.ts`, `lib/settingsTabs.ts` (`Automation` group); design records `docs/superpowers/plans/2026-07-13-routines-phase1-design.md` + `docs/superpowers/plans/2026-07-15-routine-fire-condition-design.md`. _Entry:_ Settings → Routines.

- **PTY survival architecture** — `allTerminalRefs` flattens `terminalsByWs` so every `TerminalPane` mounts at once (only the active one visible), keeping PTYs + scrollback alive across project switches. - **Git/PR refresh cadence** — Git status refreshes on workspace/mode change, on window focus, and on a 3s interval only in run/review (skipped while hidden); the diff is built only in Review. ## 2. Projects, Missions & Workspaces

Missions are the **first-level unit of intent** (build/fix/review/probe/design/perf/ops). A *code* mission (build/fix) owns exactly one **workspace**, and workspaces **are git worktrees** living as siblings of the project root under `<parent>/.octopus-worktrees/<branch>/`. The **main workspace** is special: its `worktree_path` equals the project root (git forbids the default branch checked out twice), so the project root *is* the main worktree. Isolation is a property a mission chooses along two axes — git-state (`worktree`/`readonly`/`ephemeral`/`pr`) and execution (`none`/`sandbox`/`container`/`cloud`).

### Missions (threads of intent)
- **Missions as the first-level unit** — Every workspace is born paired with a `build` mission; the pairing is idempotent, so a reused/adopted workspace keeps its mission and no workspace ever exists without one. (Design/probe missions arriving with later movements may have no worktree.) The active mission's intent surfaces as a glyph + label eyebrow above the workspace name in the ContextHeader. A partial unique index guarantees the core invariant: **never two active missions writing one checkout** — readonly missions may share a checkout freely, but their agent context is always their own. _Support:_ `missions` table + nullable `mission_id` back-refs on `runs`/`chat_threads`/`terminals` (`src-tauri/src/db.rs`); the shared `src-tauri/src/mission.rs` (`create` / `ensure_for_workspace`, one code path also used by `octopush-mcp`); commands `create_mission` / `list_missions` / `get_mission` / `update_mission` / `archive_mission`; `src/stores/missionsStore.ts`; the `ContextHeader` intent chip (`INTENT_ICON`). A one-shot, `app_meta`-gated backfill re-parents every existing workspace into a `build` mission on upgrade. _Entry:_ automatic on workspace creation; intent chip in the header.

It never fails just because the branch is in use: (1) a workspace already tracking the branch is returned (un-archived + worktree rebuilt if it was archived); (2) if the branch is already checked out somewhere — the main worktree or one made outside Octopush — that checkout is **adopted** (a workspace row is registered over it; no second worktree, since git allows a branch in only one worktree); (3) otherwise a fresh worktree is created under `.octopus-worktrees/`. Idempotent on `(project, branch)` and on the checkout path, so it never duplicates a rail row (the store also upserts by id). _Support:_ `src-tauri/src/workspace.rs` (`create`, `provision_worktree`, `reuse_or_restore`, `workspace_at_path`), `git_ops::create_worktree`/`live_worktree_on_branch`, `db::find_workspace_by_branch`/`set_workspace_worktree_path`.

### Notable implementation details
- **Worktree mechanics** — `create_worktree` returns the path it actually used and is collision-proof: the libgit2 worktree **slot name and the working-tree path are decoupled from the branch**. It refuses up front if the branch is already checked out elsewhere (git's one-checkout-per-branch rule → friendly error). Otherwise it picks a **free slot name** (reclaiming a genuinely stale `.git/worktrees/<branch>` slot in place — pruned with `valid+locked+working_tree`, then force-removed — but stepping aside to `<branch>-2`, `-3`… when that slot still backs another live worktree, e.g. one whose branch was switched: the old "directory exists" bug) and a **free path** (clearing stale leftovers, suffixing past a live worktree). It never deletes a live worktree or the project root. Creation is serialized process-wide so concurrent creates can't race. `delete_worktree` prunes the slot **by path** (slot names aren't tied to the branch anymore) and never deletes the branch.
- **DB schema — projects:** `id, name, path (UNIQUE), created_at, last_opened`, + `jira_project_key, closed_at, pinned, sort_order, tint`. **workspaces:** `id, project_id (FK CASCADE), name, task, branch, worktree_path, setup_script, status, created_at, last_active`, + `glyph, tint, test_command, linked_issue_key, issue_link_dismissed, from_branch`.
- **Customization dual-persistence** — Workspace glyph/tint → DB only. Project name/tint → both localStorage (`projectCustomizations`) and DB, with a one-time migration into the backend.
- **Tint palette** (`monogram.ts`) — 7 named tints (brass, verdigris, rouge, indigo, lavender, smoke, bone), each `{accent, bg}`.
- **URL parsing parity** — `git_url.rs` (Rust) and `parseGitUrl.ts` (TS) handle HTTPS/SCP/ssh/multi-level GitLab/self-hosted hosts identically; the TS mirror keeps clone-name auto-detect instant.

### Notable implementation details

## 4. RUN / Direct Mode (Pipeline Orchestration)

The flagship feature: compose pipelines of stages (each an AI agent with a role/model/substrate/tools/checkpoint/loop), run them, gate with checkpoints, iterate with review loops, recover from halts, and track cost/budget against an all-premium baseline.

### Pipelines — templates & authoring model
- **Pipeline templates** — A named, reusable graph of stages; each stage is one AI agent. Stored as templates, instantiated per-run. _Support:_ `pipelines` + `pipeline_stages` tables; `list_pipelines`, `get_pipeline`.
- **5 built-in pipelines (seeded, idempotent)** — `db.rs::seed_builtin_pipelines`: **Feature Factory** (plan→plan_review→implement✓→code_review✓ loop→test✓), **Bugfix relay** (repro→fix✓→verify✓ loop), **Plan & review** (plan→critique→refine✓, linear), **Ship it** (plan→implement✓→code_review✓ gated loop→test✓→**pull_request**✓ cli — the issue-to-PR crew), **Claude Code build** (plan(api)→implement(cli)✓→code_review(cli)✓ loop→test(cli)✓). A one-shot UPDATE retrofits loop config onto pre-loop builtin installs.
- **Custom pipelines (create / fork / update)** — `save_pipeline` has 3 modes: create new; **fork a builtin to a new custom copy** (builtin never mutated); update a custom in place (transactional). _Support:_ `save_pipeline`; `pipelineStore.save`. _Entry:_ builder "Save as my copy" / "Save pipeline".
- **Delete pipeline** — Custom only; builtins protected. _Support:_ `delete_pipeline`. _Entry:_ builder footer two-step.
- **Pipeline validation** — `validate_pipeline_stages`: ≥1 stage; role must exist; substrate ∈ {api,cli}; non-empty model; `max_iterations` 1..=100; parents form an acyclic DAG; tools a subset of `KNOWN_TOOLS`; instructions ≤ 8000 chars; loop config rules (target requires `can_loop` role, flow-ancestor, mode ∈ {gated,auto}). _Support:_ `db.rs`.

### Node-based visual pipeline builder

### Stage substrates & execution
- **API substrate (in-app agentic loop)** — `ApiRunner` runs a stage through `run_agentic_loop`: resolves provider, composes system prompt, builds the input dossier, runs a tool-use loop bounded by `max_iterations`, returns exact token usage + cost. _Support:_ `orchestrator/runner.rs`, `agentic.rs`.
- **Per-stage reasoning effort (the cost/quality lever)** — Each stage carries an optional `effort` (`Effort` enum: `low·medium·high·xhigh·max`; `null`/absent = **off**, no thinking — today's default). It's **editor-set** in the pipeline builder (the Stage inspector's Reasoning control) — and equally **authorable over `octopush-mcp`** (`create_pipeline`/`update_pipeline`, documented in `describe_pipeline_schema`) — persisted on `pipeline_stages`, copied to `run_stages` at `create_run`, and read into `StageSpec.effort` → `LlmRequest.effort`. It's the atomic knob the cost model rests on: a cheap model thinking deeply on synthesis stages, barely thinking on mechanical ones. Applied on the **API substrate only** — a CLI stage's agent manages its own reasoning, so the control is disabled and no effort pill renders for CLI stages. **Model-capability mapping** (pure, unit-tested, Anthropic): effort-capable models (Opus 4.x, Sonnet 5/4.6, Fable 5, unknown → the safe majority) get `output_config.effort:"<level>"` + `thinking:{type:"adaptive"}`, where the **level is clamped per model** (`effective_effort_level`) so we never 400 — only Opus 4.7/4.8, Sonnet 5, Fable 5, Mythos 5 accept `xhigh`; Opus 4.6 / Sonnet 4.6 (the default) fold `xhigh`→`high` but keep `max`; Opus 4.5 / unknown fold both `xhigh` and `max`→`high`. The thinking-but-not-effort models (`haiku-4-5`, `sonnet-4-5`, `sonnet-4-0`) instead get `thinking:{type:"enabled", budget_tokens:N}` (N = low 4096 / med 8192 / high 16384 / xhigh·max 24576, clamped `[1024, max_tokens-1]`) with **no** `output_config` (they 400 on it). Classification is a strict **allowlist** (`is_effort_model` / `is_budget_model`): a legacy (`claude-3-5-*`) or unknown id matches neither, so it gets **no thinking params at all** (effort silently ignored — never 400 a non-thinker). `temperature` is never sent (current models 400 on it with thinking). Effort also raises the `max_tokens` floor (`max_tokens_for`: none/low/med 32768 · high 48000 · xhigh/max 64000) so deep thinking doesn't truncate the answer. With thinking + tool use, a completed assistant turn's **full content-block array is captured verbatim and replayed in order** (`LlmResponse.raw_content` → `AssistantWithTools.raw`) so signed thinking survives and interleaved thinking/tool_use ordering is preserved (reordering 400s); the max_tokens-truncation retry path deliberately replays with **empty** raw (a cut-off turn can carry an unsigned partial thinking block) and rebuilds from text + tool_uses. OpenAI-compat maps `effort` → `reasoning_effort` (low/med/high; xhigh·max fold to high) **only on reasoning models** (o1/o3/o4, GPT-5 — omitted on gpt-4o), and picks the token-cap key by class: reasoning models use `max_completion_tokens` (they 400 on `max_tokens`), chat models use `max_tokens`; both pass through unclamped. _Support:_ `providers/mod.rs` (`Effort`, `raw_content`), `providers/anthropic.rs` (`is_effort_model`/`is_budget_model`/`effective_effort_level`/`thinking_json`), `providers/openai_compat.rs` (`openai_supports_reasoning`), `orchestrator/agentic.rs` (`max_tokens_for`), `db.rs` (`effort` column on `pipeline_stages`/`run_stages`), `builder/StageInspector.tsx`. Design record: `docs/superpowers/plans/2026-07-14-per-stage-effort-design.md`.
- **Automatic model escalation (per stage)** — A stage can carry an optional **escalation policy** — `escalate_model` (a stronger model to retry with) and/or `escalate_effort` (a higher effort to bump to on the retry, API-only like base effort). A stage **has a policy iff either field is set**. It's **editor-set** in the Stage inspector's **"Escalate on failure"** control (an escalate-model picker with a "— none —" clear + an escalate-effort segmented control, disabled for CLI with the same note as the base effort) — and equally **authorable over `octopush-mcp`** (`create_pipeline`/`update_pipeline`, documented in `describe_pipeline_schema`) — persisted on `pipeline_stages`, copied to `run_stages` at `create_run`. **Trigger = a stage FAILS** (`StageStatus::Failed` — includes the agentic loop exhausting `max_iterations` unfinished): the drive's Failed arm calls `try_escalate`, which retries the stage **once** at the strong tier before halting — "pay the expensive tier on a *proven* failure, not by default". `try_escalate` re-reads the fresh stage row (the Failed arm's clone is pre-run state) and mirrors the re-run reset the reject/loop-back paths use: it **archives** the failed base-tier attempt to `stage_iterations`, **retires** its spend (so `recompute_run_cost` keeps counting it and the budget gate stays honest), **resets** the stage to pending **preserving** the existing feedback (and loop counter/session), sets the sticky `escalated` flag, and appends a live journal notice (`↑ Escalating to {model} after a failed attempt`); the drive `continue`s and re-runs the now-pending stage. It escalates only when the tier would ACTUALLY change (`escalate_model` ≠ base, or an API-substrate `escalate_effort` ≠ base) — a CLI effort-only or same-model policy halts instead of burning an identical retry. **Spec resolution** (`run_stage_once`): an escalated stage resolves `agent_model` = `escalate_model` (falling back to the base when only effort was bumped) and `effort` = `escalate_effort.or(base)` — the **base fields stay on the row** for history/display. **Bounded to exactly one escalation**: a second failure hits `try_escalate` → already-escalated → halt. Because the flag is sticky, any later re-run (a review loop-back, a manual reject) also uses the strong tier for free. **`escalate_model` applies to BOTH substrates** (a plain model swap on the spec; the CLI runner reads `agent_model` too); **`escalate_effort` is API-only**. **A blocked stage (`ask_director` → `awaiting_checkpoint`) and an aborted run NEVER escalate** — only the Failed arm calls `try_escalate`. **Budget/detached-compatible for free**: the per-run budget gate runs before a stage starts (an escalated retry over budget pauses for budget), and escalation converts a would-be-halt into a `continue` within the same drive segment, so a detached worker re-runs the escalated stage in-segment exactly as the in-process app does. A run-stage that escalated shows a brass `↑ {model}` badge on its `RunFlow` meta line; the authoring surfaces (`StageNode`, `StageFlow`) show the policy as a brass `↑ {model}` hint. **Ungated** (a reliability property, like per-stage effort and the escape valve). _Support:_ `orchestrator/mod.rs` (`try_escalate`, `drive_inner` Failed arm, `run_stage_once` spec resolution), `db.rs` (`escalate_model`/`escalate_effort` on `pipeline_stages`+`run_stages`, `escalated` on `run_stages`, `set_run_stage_escalated`, `RunStageRow`/`PipelineStageRow`/`StageDraft`), `builder/StageInspector.tsx` + `builder/graph.ts`, `RunFlow.tsx`/`direct/StageFlow.tsx`/`builder/StageNode.tsx` badges, `lib/ipc.ts`. Design record: `docs/superpowers/plans/2026-07-14-auto-escalation-design.md`.
- **Agentic loop internals** — Per-stage tool allowlist; transient-error retry with backoff + journal narration; proactive rate-limit throttle; tool results capped at 24KB fed back; cancel flag checked each iteration; an unfinished loop is a **failure** (usage preserved). _Support:_ `agentic.rs`.
- **CLI substrate (headless Claude Code)** — `CliRunner` spawns `claude -p --output-format stream-json --verbose --model … --append-system-prompt … --permission-mode bypassPermissions --max-turns N` in the worktree, streams NDJSON, parses the terminal `result` event. _Support:_ `cli_runner.rs`.
- **CLI environment & PATH resolution** — Captures the user's full login+interactive shell env once (`SHELL -lic 'env -0'`) so a GUI-launched app inherits PATH/exports; resolves the `claude` executable across login PATH ∪ inherited ∪ common dirs. _Support:_ `login_shell_env`, `resolved_cli_path`.
- **CLI timeouts** — Idle (300s silence) + absolute (3600s) with distinct honest messages; stderr drained concurrently. _Support:_ `cli_runner.rs`.
- **Uniform runner abstraction** — Both substrates implement `AgentRunner::run(stage, input, ctx) -> StageOutcome`; the orchestrator never branches on substrate. _Support:_ `runner.rs`.
- **Stage input dossier (cross-stage handoff)** — Each stage receives the run task, a one-line pipeline breadcrumb, and the **freshest artifact of each kind** from feeding stages (transitive flow-ancestors; sibling branches don't leak), each capped at ~16K chars. **Reviewers see the actual code:** when feeding artifacts ref the worktree, the **live diff (staged + unstaged — a CLI agent habitually `git add`s) is captured and embedded** in the dossier between `BEGIN/END GIT DIFF` markers (not a ``` fence — a diff of a markdown file contains fence lines), capped head+tail **with an explicit truncation notice** and a shared-workspace caveat — so `code_review`/`verify`/`test` judge the real changes, not the implementer's prose summary. Best-effort: capture failure/empty diff falls back to the "inspect with your tools" hint; skipped for CLI-resume stages (input discarded). _Support:_ `orchestrator::full_worktree_diff` (shared with the stage diff snapshot), `assemble_stage_input`, `StageInput.worktree_diff`, `user_input_for`.

- **Ship a GitHub issue** — In the launcher's brief card: a quiet `Github` entry ("ship an issue") opens a picker of the repo's **open GitHub issues** (`gh issue list` in the login shell, mirroring `list_prs`). **Preflight first** (`github_ship_readiness`): no github.com origin → honest dead end; `gh` unauthenticated → "run `gh auth login`" guidance — the crew's `pull_request` stage would otherwise fail mid-run AFTER building the change. Picking an issue **folds it into the task** (`composeIssueTask`: `Ship GitHub issue #N — title` + body capped at 4KB with an explicit truncation note + `include "Closes #N"` instruction — the task reaches every stage's prompt, so the PR stage closes the issue) and **selects the "Ship it" builtin**. Deliberately does NOT touch `linked_issue_key` (Jira-semantic everywhere — a `#N` there would be fed to the Jira client). All issue values render as inert text. _Support:_ `GithubIssuePicker.tsx`, `lib/shipIssue.ts`, commands `list_github_issues`/`github_ship_readiness` (+ `github.rs::issues_from_json`/`ShipReadiness`). _Entry:_ Direct launcher → brief card → "ship an issue".

### Runs — lifecycle & control
- **Create run (draft)** — Copies pipeline_stages into a private `run_stages` copy, applies positional `(position, model)` overrides, inserts as `draft`. _Support:_ `create_run`; `runsStore.begin`.
- **Start run (background drive)** — Spawns the drive as a `tokio` task; persists optional budget. _Support:_ `start_run` → `orch.start_run`.
- **Run status lifecycle** — `draft → running → paused → completed | aborted | failed`. _Support:_ `RunStatus`; status meta in `runStatus.ts`.
- **Stage status lifecycle** — `pending → running → {awaiting_checkpoint | done | failed}`; the drive picks the lowest-position non-`done` stage. _Support:_ `StageStatus`; `runStatus.ts`.
- **Director pause** — Parks the next pending stage at the boundary. _Support:_ `request_run_pause`; a quiet `Pause` `IconButton` in the run header (`DirectCanvas`) — always mounted, opacity-faded + disabled while a control isn't applicable.
- **Stop current stage** — Real cancellation: API loop halts at next check, CLI kills the child; stage lands `failed` in halt-recovery. _Support:_ `stop_stage`; a `CircleStop` `IconButton` in the run header (`DirectCanvas`).
- **Abort run** — Marks `aborted` AND kills the in-flight stage. _Support:_ `abort_run`.
- **One executing run per workspace** — `start_run` rejects a concurrent running/paused run in the same workspace (runs share one worktree); UI gate disables "Begin the run". _Support:_ `has_concurrent_run`; `runsStore.hasExecutingRun`.
- **Interrupted-run recovery (startup)** — Stages stuck `running` after a crash are stamped `failed` (`INTERRUPTED_STAGE_ERROR`); runs re-parked `paused`; the amber Resume affordance keys off the "interrupted" prefix. _Support:_ `db.recover_interrupted_runs`.

### Checkpoints, gates & review loops
- **Checkpoint gates** — A `checkpoint=true` stage parks `awaiting_checkpoint` + run `paused` on completion. _Support:_ `checkpoint` column; `drive_inner`.
- **Review loops — two modes** — A review stage (`can_loop`) can route work back to a `loop_target`. **Gated:** the run pauses; the human chooses Send-back vs Approve-anyway. **Auto:** the orchestrator loops target↔review automatically on a parsed `VERDICT: CHANGES_REQUESTED` until it passes or hits the cap. _Support:_ `LoopMode`; loop columns; `parse_verdict`.
- **Loop-back mechanics** — Resets the contiguous `[target..=review]` range (restricted to flow-ancestors) to `pending`, forwards the reviewer's findings as feedback, archives erased attempts, retires their cost, bumps the loop counter, re-drives. Cap reached → escalates to a checkpoint. _Support:_ `loop_back`, `db.increment_loop_iteration`, `db.retire_stage_cost`.
- **Checkpoint actions (full vocabulary)** — `resolve_checkpoint`: **Approve** (accept & continue; a failed stage's partial work is accepted via a synthesized artifact), **Reject** {feedback, model_override, max_turns_override} (re-run the single stage), **SendBack** {feedback} (loop the review back to its target), **Resume** {max_turns_override} (recover a halted stage; CLI `--resume`s the same session), **Discard** (revert the worktree to the failed stage's baseline), **Edit** (artifact edited out-of-band; continue), **AnswerBlocker** {answers} (answer a stage that parked itself via `ask_director` — see the escape valve below), **Abort**. _Support:_ `CheckpointAction`; `runsStore.resolve`; `RunControlBar`.
- **Escape valve — ask the director (`ask_director`)** — A DIRECT stage that hits a decision only the director can make (a genuine ambiguity, a missing spec/credential, contradictory requirements) **stops and asks instead of guessing**: it calls the `ask_director` tool with a one-line `summary` and structured `questions`, each carrying a **recommended default**. The tool is appended to the agentic loop **after** the per-stage allowlist filter (so it survives a review stage's read-only tools) and is **DIRECT-only** — never in TALK's shared `build_llm_tools`. On the call the loop **stops** (asking supersedes acting: any co-requested tool_use is discarded; the `BlockedAsk`/`BlockedQuestion` structs are tolerant — every field defaults and the underscore fields accept both camelCase and snake_case, so a well-formed multi-question ask never collapses; only genuinely garbage input degrades, and even then it salvages **all** array elements) and the runner returns a **blocked outcome**; the drive persists the questions to `run_stages.blocked_questions`, parks the stage `awaiting_checkpoint`, and fires the **same** `emit_checkpoint(reason:"decision")` a gate uses — so the crew "needs you" notification and the attention beacon fire unchanged (no new `StageStatus`; a block is just another checkpoint, distinguished by `blocked_questions IS NOT NULL`). The director answers via **`answer_blocker(run_id, stage_id, answers)`**: the decisions (each paired with its recommended default) become the stage's re-run **feedback**, `blocked_questions` clears, the asking spend is retired, and the stage **re-runs with the answers injected** (reusing the reject→feedback→re-run substrate — a block resolves by re-running, not a mid-loop resume). A block is **answer-only**: `apply_checkpoint_action` refuses Approve/Reject/SendBack on a stage with `blocked_questions` (only AnswerBlocker, or Abort, proceed) and refuses AnswerBlocker on a non-block — a stale click can't mark a block done with an empty hand-off. A block also stamps `started_at`, so it is **not** treated as an editable un-begun park (`update_run_stage`/`StageFocus` require `started_at IS NULL`, not just "no artifact"). **Recommended-default fast path:** the answer UI pre-fills each input with the default, so accepting is one click. **API-substrate only** (CLI stages own their own interaction model): the carve-out is a separate `ASK_DIRECTOR_CLAUSE` appended by `compose_system_prompt(can_ask_director=true)` for API stages **only** — CLI stages keep the strict never-ask preamble intact (they have no such tool). **Ungated** (core trust, like crew notifications). _Support:_ `agentic.rs` (`ask_director_tool`/`parse_ask_director`, `AgenticResult.blocked`), `types.rs` (tolerant `BlockedAsk`/`BlockedQuestion`, `StageOutcome.blocked`, `CheckpointAction::AnswerBlocker`), `runner.rs`, `orchestrator/mod.rs` (`run_stage_once` block-park, `format_blocker_feedback`, `apply_checkpoint_action` answer-only guard), `db.rs` (`run_stages.blocked_questions` + `set_run_stage_blocked`; `update_run_stage` un-begun guard), `commands::answer_blocker`, `roles.rs` (`ASK_DIRECTOR_CLAUSE` + `compose_system_prompt(can_ask_director)`); frontend `RunControlBar` `BlockedAskBar`, `MissionControl` block indicator, `StageFocus` un-begun guard, `runsStore.answerBlocker`, `ipc.answerBlocker`/`BlockedAsk`; design record `docs/superpowers/plans/2026-07-14-escape-valve-design.md`. _Entry:_ automatic when a stage blocks; answered from the decision bar.
- **Decision bar UI (state-adaptive)** — `RunControlBar.tsx` switches on status: TerminalBar ("Run it again"), **DraftBar** (a STAGED run — e.g. authored by `octopush-mcp` `create_run` — mono `draft` word, serif-phrase primary CTA **"Begin this run"** → `runsStore.start` → `start_run(runId, null)` + detail refresh, and a **discard** control via `abort_run`; a refused start (quota/concurrency gate) shows the upgrade sheet and leaves the draft intact — never a dead end), DecisionBar (checkpoint/halt, with Resume/Re-run turn-budget Stepper, Accept-partial, Discard-with-confirm, "why this halted" disclosure, feedback editor for reject/send-back, loop-state banner), **BlockedAskBar** (the escape valve: when a parked stage carries `blockedQuestions`, the decision gives way to **"The crew asked you"** — one input per question pre-filled with its recommended default, plus **"Send answers"** and **"Accept all defaults"**). While a run is simply running the bar renders **nothing** — pause/stop/abort live as the header's `IconButton`s instead (see Director controls above), and the single brass beacon (`lib/beacon.ts` `beaconAnchor`) sits on the DraftBar/DecisionBar's primary CTA or, mid-run, on the running stage's card in `RunFlow` — never on this bar. A started draft behaves exactly like a launcher-started run (checkpoints, gates, ledger); `hasExecutingRun` still counts only running/paused. _Entry:_ bottom command bar.

The reveal-after-finish drawer is labelled plain "work journal · N" (no `§`). _Support:_ `StageFocus.tsx`; `lib/roleIcons.ts`.
- **Journal persistence** — `PersistingSink` mirrors every `run://log` entry into `stage_log` (reset events → marker rows) so journals survive reloads and loop-backs. _Support:_ `orchestrator/persist.rs`; `get_stage_log`.

### Iteration history & diff snapshots

- **Focus-follows-action** — A pinned running stage that finishes auto-clears the pin so focus returns to the active stage. _Support:_ `DirectCanvas`.

### Director hot controls (mid-run redirection)
- **Hot-edit a pending stage** — Gate toggle (`⟜ gate` TogglePill: gated ↔ automatic hand-off) and an **"Edit stage"** modal (instructions, model, turn budget, loop mode for looping reviews) on any stage the run hasn't reached — live, no pipeline recreation, no restart, no reload. Edits touch only the run's own `run_stages` row (the template is never modified) and are honored when the stage builds its spec. _Support:_ `update_run_stage` (commands.rs → `db::update_run_stage`, guards: run non-terminal, stage un-begun); `runsStore.updateStage` (optimistic + `refreshDetail` reconcile); StageFocus header controls.
- **Field edits on a parked, un-begun stage** — A stage parked **before it began** (a budget park or a director pause, which hold the next stage with no work done) still takes field edits; the park itself is released via approve/reject, so toggling the gate while parked is rejected with a clear error. "No artifact yet" is the no-work-done discriminator: pre-work parks produce nothing, while a checkpoint **gate** park holds a *finished* stage's hand-off (artifact present) and is redirected via the decision bar or a re-run instead. _Support:_ `db::update_run_stage` `parked_unbegun` branch; StageFocus `fieldsEditable`.
- **Re-run from here** — A finished (done/failed) stage — and everything downstream — re-runs **in place**: downstream checkpoint parks are invalidated and archived, CLI sessions are never resumed (`session_id`/`resume_pending` cleared), loop counters restart, spend is retired onto the run (append, never reset). Offered on paused/completed/**failed** runs and on running runs **parked** at a checkpoint (`runBlocked`); guarded by an inline "discards downstream" confirm; mutually exclusive with concurrent resolve/rerun via the run's active-slot claim. _Support:_ `rerun_from_stage` → `prepare_rerun`/`resume_claimed_drive`; StageFocus `rerunnable` + confirm Reveal.
- **Re-run after changes** — A finished stage's Edit button becomes **"Edit & re-run stage"**: the same modal (plus the gate toggle as a field) whose primary action **"Save & re-run from here"** ships the director's patch WITH the re-run, atomically — validated before anything resets (a bad patch leaves the run untouched), applied once the stage is back to pending, honored by the re-driven spec. _Support:_ `StageRerunPatch` riding `rerun_from_stage`; `db::validate_stage_patch_fields`/`apply_run_stage_patch`; shared wire encoding `stagePatchArgs` in `ipc.ts`.

- **Live cost & all-premium baseline** — Per-stage actual cost accumulated; baseline re-prices the same tokens at the **premium reference model** (highest blended price among enabled providers; per-pipeline overridable). ### Notable implementation details
- **Orchestrator architecture** — `Orchestrator` (`orchestrator/mod.rs`) drives runs as background tokio tasks, one stage at a time, pausing at checkpoints; enforces a single active drive per run, holds per-run cancel flags + pause requests, routes every emit through a `PersistingSink`. The `EventSink` trait decouples it from Tauri for testability.
- **Drive loop** — Pick lowest non-`done` stage; complete the run if none; check budget gate then director-pause; run the stage; on a block (`ask_director`) → paused+checkpoint carrying the questions; on Failed → **try one automatic escalation** (`try_escalate`: a stage with an escalation policy that hasn't used it retries once at the strong tier, else) paused+checkpoint; on Done → auto-loop verdict decision before the gated pause. Aborts mid-stage always win.
- **DB tables** — `pipelines`, `pipeline_stages` (+ pos/parents/tools/custom_name/instructions/loop/max_iterations/effort/escalate_model/escalate_effort), `runs` (+ cost/baseline/reference_model/retired/budget/linked_issue_key), `run_stages` (private per-run copy + status/tokens/cost/artifact/feedback/error/loop/diff_snapshot/session_id/resume_pending/baseline_commit/effort/blocked_questions/escalate_model/escalate_effort/escalated), `roles`, `stage_log`, `stage_iterations`, `run_events`. The `effort` column (nullable TEXT, the `Effort` token; NULL = off) is additive on `pipeline_stages` and `run_stages`; `blocked_questions` (nullable TEXT, JSON `BlockedAsk`) is additive on `run_stages` (the `ask_director` escape valve); the escalation policy — `escalate_model` (nullable TEXT), `escalate_effort` (nullable TEXT) — is additive on both, and `escalated` (INTEGER DEFAULT 0, sticky run-state) is additive on `run_stages`.
- **Tauri events** — `run://stage-update`, `run://cost`, `run://checkpoint`, `run://error`, `run://log`. Subscribed once in `runsStore`.
- **Worktree as the blackboard** — Stages never commit (handoff = worktree + structured artifact); all runs in a workspace share the one worktree, which is why only one executes at a time. A completed run leaves changes uncommitted for Review mode.
- **Shared micro-track primitive** — `direct/StageDots.tsx` is the universal stage-dot track: "status" tone colors by run state, "shape" tone is the neutral launcher-ticket variant (every dot sage, only the gate ring carries meaning). Now consumed by `PipelineTicket`'s ticket shape line.

## 6. Git & GitHub

### Git status & change detection
- **Working-tree status (libgit2)** — Branch, ahead/behind, has-upstream, conflict count, in-progress operation, and per-file `FileChange` (`path`, `status`, `staged`/`unstaged`/`conflicted` from libgit2 status flags; untracked included). _Support:_ `git_ops::status_files`/`get_status`; `get_git_status` (splits the cheap file-walk from the slow ahead/behind). _Entry:_ `ChangesPanel` polls every 5s.
- **Ahead/behind vs upstream (timed)** — Graph walk under a 3s timeout; on timeout sets `aheadBehindKnown=false` so the UI hides ↑/↓ rather than show a misleading 0. _Support:_ `git_ops::ahead_behind`.
- **In-progress operation detection** — Maps `RepositoryState` to `merge`/`rebase`/`cherry-pick`. _Support:_ `git_ops::operation_state`. _Entry:_ drives the conflict section + Continue/Abort.
- **Fast dirty check (rail)** — `is_dirty` (no untracked recursion) + `dirty_ahead_behind`. _Support:_ `git_ops`.
- **Per-workspace git summary (batch)** — One `WorkspaceGitSummary{dirty, ahead, behind}` per worktree; per-worktree timeouts; a single unreadable worktree defaults to clean. _Support:_ `workspaces_git_summary`. _Entry:_ rail chips.
- **Branch listing** — `{local, remote}` with the default (HEAD) branch promoted first; remote as `origin/dev`, `*/HEAD` excluded. _Support:_ `list_branches`. _Entry:_ GitOpsMenu, base picker.

### Diff reading (git operations side)

- **Commit log (paginated)** — First-parent revwalk, `skip`/`limit`; each `CommitInfo{sha, shaShort, summary, authorName, timestampMs}`. _Support:_ `git_log`. _Entry:_ `HistoryModal` (page 50).

### Staging & unstaging

### Commit & amend
- **Commit staged changes** — `git commit -m '<escaped>'` through the user's **login shell** so gitconfig identity, hooks, and signing behave like the terminal; rejects empty; returns the new short SHA. _Support:_ `commit_changes` (holds git_lock). _Entry:_ Commit button / `c` shortcut.

### GitHub & pull requests

### Notable implementation details
- **git2 vs shell-out boundary** — Read/inspect ops use libgit2 (status, diffs, log, blame, branch/stash list, create_branch, tags, worktree lifecycle). Mutating/auth-sensitive ops shell out so they inherit the user's full git environment: commit, amend, push, fetch, pull, switch, cherry-pick, continue/abort, reset, clean, stage/unstage, hunk apply, conflict take, discard, PR-branch fetch.
- **Auth model** — Push/pull/fetch/switch/commit/cherry-pick/continue/abort run via the user's **login shell** (`$SHELL -l -c`) so SSH agents, the osxkeychain helper, and `~/.gitconfig` behave as in a terminal. Clone uses a `GIT_ASKPASS` tempfile reading the saved PAT. PR lookups use the saved github.com PAT or `GITHUB_TOKEN`/`GH_TOKEN`, preferring `gh`.
- **The git lock** — `git_lock.rs` keeps a global per-path async mutex so two mutating ops on the same worktree can't interleave; read-only commands take no lock.
- **PR state classification** lives entirely in `github.rs`; the frontend only maps `PrState` → color/glyph. The pure classifiers (`classify_pull`, `classify_continue`, `friendly_switch_error`, `pr_from_json`, `parse_git_url`) are heavily unit-tested.

## 7. Terminals, PTY Daemon & Sessions

### PTY Daemon — out-of-process architecture
- **Standalone PTY daemon (`octopush-pty-server`)** — A separate Rust binary that owns ALL pseudo-terminals so shells survive Octopush restarts/auto-updates. _Support:_ `src-tauri/src/bin/octopush-pty-server/*`; bundled as a sibling binary. _Entry:_ auto-spawned on app launch by `pty_daemon::ensure_daemon_running()`.
- **Detached spawn** — Launched with `setsid()` + null stdio so it escapes Octopush's process group and is adopted by launchd. _Support:_ `pty_daemon.rs::spawn_detached`.
- **Binary resolution** — Sibling of `current_exe()`, then `$PATH`, then `target/debug/…`. _Support:_ `resolve_daemon_binary`.
- **Unix domain socket** — Newline-delimited JSON-RPC at `~/.octopush/pty-server.sock` (chmod 0700). _Support:_ `pty_daemon.rs`, `main.rs`.
- **PID-file double-start protection** — `~/.octopush/pty-server.pid`; a second instance verifies the PID is alive AND is an `octopush-pty` process before exiting. _Support:_ `main.rs::acquire_pid_file`.
- **Protocol version handshake / stale-daemon replacement** — Reuses a running daemon across version bumps if `protocol_version` matches (`EXPECTED_PROTOCOL_VERSION = 2`); on mismatch SIGTERMs the old daemon and spawns the new one. _Support:_ `query_daemon_protocol`, `kill_existing_daemon`.
- **Idle auto-exit** — Exits when zero live PTYs AND zero clients for `OCTOPUSH_PTY_AUTO_EXIT_SECS` (default 3600s). _Support:_ `server.rs` idle thread.
- **Clean shutdown + crash recovery** — SIGTERM/SIGINT removes pid/socket; a stale socket is unlinked on startup; orphan scrollback logs swept. _Support:_ `main.rs`, `storage.rs`.
- **fd-limit raising** — Raises `RLIMIT_NOFILE` so terminal spawns don't fail with EMFILE. _Support:_ `main.rs::raise_fd_limit`.
- **Daemon log** — `~/.octopush/pty-server.log`, rotated at 5 MiB. _Support:_ `storage.rs`.

### Daemon socket protocol
- **Request methods** — `list_terminals`, `spawn`, `attach{since_seq?}`, `detach`, `write`, `resize`, `kill{signal?}`, `remove`, `shutdown`, `version`. _Support:_ `protocol.rs`, `server.rs::dispatch`.
- **Streaming events** — `data{id,seq,bytes}`, `exit{id,code?}`, `error{id,message}`, `attention{id}`. _Support:_ `protocol.rs::Event`.
- **Per-connection model** — Each connection on its own thread; a writer thread drains an unbounded channel; on disconnect, only that connection's attachments detach. _Support:_ `server.rs::handle_connection`.

### Scrollback persistence & replay
- **Dual buffering: ring + disk** — Each chunk gets a monotonic `seq`, stored in a 64 KiB ring buffer AND appended to a disk log. _Support:_ `session.rs::push_output`.
- **Disk-backed log** — `~/.octopush/pty-state/<id>.log`, capped at 1 MiB by rotation. _Support:_ `storage.rs`.
- **Scrollback replay on attach** — Atomic snapshot under the lock: full disk log replay if `since_seq` is older than the ring, else ring backlog; live data streams after with no overlap. _Support:_ `server.rs::cmd_attach`.
- **Sessions outlive their shells** — A session is kept after its shell exits (for replay) but its fds are released eagerly. _Support:_ `server.rs`, `session.rs`.

### Spawning, attaching & terminal lifecycle (daemon side)
- **Spawn a shell** — Opens a PTY via `portable_pty`, launches `$SHELL` (or `/bin/zsh`) as a login shell in `cwd` with a curated env allowlist + `TERM`/`COLORTERM`, then merges caller env. _Support:_ `server.rs::cmd_spawn`.
- **Duplicate-id / respawn-in-place** — A spawn for a live id is refused; an EXITED id is replaced in place (log deleted, seqs restart). _Support:_ `cmd_spawn`.
- **Write / resize / kill / remove** — base64 write; PtySize resize; SIGTERM/SIGKILL kill (session retained); `remove` SIGKILLs the process group + deletes the log. _Support:_ `cmd_write`/`cmd_resize`/`cmd_kill`/`cmd_remove`.

### Attention detection (daemon-side)
- **"Waiting for input" detection** — A 1s thread fires `Event::Attention` on (A) command-finished (fg pgroup returns to the shell) or (B) TUI-parked (fg child <1% CPU for two ticks + ≥2s quiet). _Support:_ `server.rs` attention thread, `session.rs::check_attention`.
- **False-positive gating** — Requires ≥200 bytes since last alert; a 3s post-attach grace; baselines wiped on attach. _Support:_ `session.rs`.

### App-side daemon client
- **`DaemonClient`** — One `UnixStream`; a reader thread multiplexes responses (per-`reqid` waiters, 5s timeout) and per-terminal event subscribers. _Support:_ `pty_client.rs`.
- **Single-subscriber-per-terminal routing** — A re-attach replaces the prior subscriber so each byte renders once. _Support:_ `pty_client.rs::attach`.
- **Self-healing reconnect** — On socket EOF the next call re-runs `ensure_daemon_running()` and swaps in a fresh socket. _Support:_ `try_reconnect`.
- **Stub client (degraded mode)** — If the daemon binary is absent, a pre-broken stub lets Octopush still launch and heals on first reconnect. _Support:_ `pty_client.rs::stub`.
- **`PtyManager` + Tauri event bridge** — Forwards `TermEvent`s as `pty://data`/`pty://exit`/`pty://attention`/`pty://reattached`. _Support:_ `pty_manager.rs`.
- **`spawn_or_attach` + `OutputHook`** — Reattaches a running id (replays scrollback) or spawns fresh; an optional `OutputHook` runs per chunk for token scanning. _Support:_ `pty_manager.rs`.

### Backend commands & session model
- **Session commands** — `create_session` (UUID, ContextGuard env, token hook, mark Active), `list_sessions`, `write_to_session`/`write_text_to_session`, `resize_session`, `kill_session`, `delete_session`. _Support:_ `commands.rs`.
- **Terminal commands (Run mode)** — `list_terminals`, `create_terminal`, `rename_terminal`, `delete_terminal` (DB-first, then daemon `remove`). _Support:_ `commands.rs`; `terminals` table.
- **PTY-bridge commands** — `spawn_or_attach_terminal` → `{mode:"Spawned"|"Reattached", pid?}`; `list_pty_sessions` (swallows daemon errors → empty for graceful degradation). _Support:_ `commands.rs`.
- **Session record & statuses** — `Session{id, name, color, icon, project_root, agent(provider/model/…), token_budget, tokens_used, status, context_files, tags, …}`; status `Active|Idle|Paused|Completed|Error`. _Support:_ `session.rs`.

### Session restore & env injection
- **`restore_active_sessions`** — On `.setup()`, re-spawns a PTY for each `Active`/`Idle` session, re-applying ContextGuard env + the token hook; marks `Error` on failure. _Support:_ `lib.rs`.
- **`ContextGuard::auto_configure`** — Produces the isolated PTY env: `HISTFILE` (per-session isolated history), `OCTOPUSH_PROJECT_TYPE` (Rust/Node/Python/Go/Java/Ruby/Unknown), `OCTOPUSH_GIT_BRANCH`. _Support:_ `context_guard.rs`.
- **Context-file detection** — Detects `CLAUDE.md`, `.claude/settings.json`, `AGENTS.md`, `GEMINI.md`, `.cursorrules`, `.github/copilot-instructions.md`, `CONVENTIONS.md` (stored on the session, not exported). _Support:_ `context_guard.rs`.
- **`OCTOPUSH_MODEL` injection** — `create_session` injects the model id so CLI agents can read it. _Support:_ `commands.rs`.
- **Token-scan output hook** — The per-chunk `OutputHook` runs `token_engine::scan_pty_output` → `TokenEngine.record`. _Support:_ `commands.rs`, `lib.rs`.

### xterm.js front-end (TerminalPane)
- **Terminal instance** — `@xterm/xterm` with cursorBlink, JetBrains Mono, 10000-line scrollback. Theme is built live from the active Octopush theme's tokens (`getXtermTheme()`), not a fixed palette, so RUN & TALK terminals stay legible under every theme including the light "vellum" theme — reassigned on the `octo:theme` event without remounting. _Support:_ `TerminalPane.tsx`, `TerminalView.tsx`, `lib/xtermTheme.ts`, `stores/themeStore.ts`.
- **FitAddon + ResizeObserver** — Fits xterm; debounced resize sends `resizeSession` only on actual change; refits on visible toggle + `layoutVersion` bumps. _Support:_ `TerminalPane.tsx`.
- **WebLinksAddon + file-path link provider** — Linkifies URLs; path-shaped tokens (with `:line[:col]`) become clickable links routing to Review/editor. _Support:_ `TerminalPane.tsx`.
- **Clipboard bridge** — Cmd-C / Ctrl-Shift-C / right-click-with-selection copy via `term.getSelection()`. _Support:_ `TerminalPane.tsx`.
- **Reattach SIGWINCH "wiggle"** — On `pty://reattached`, briefly resizes cols to force alt-screen TUIs (Claude Code/vim/htop) to redraw. _Support:_ `TerminalPane.tsx`.
- **Attention surfacing** — `pty://attention` and xterm's `onBell` ping `attentionStore` only when the pane is NOT visible. _Support:_ `TerminalPane.tsx`.
- **PTY survives unmount** — Cleanup disposes xterm but never kills the PTY; only the × button triggers `deleteTerminal`. App mounts a `TerminalPane` for EVERY `(workspace, terminal)` pair. _Support:_ `TerminalPane.tsx`, `App.tsx`.

### UI surfaces
- **CompanionTerminals** — Run-mode terminal list with status dots, `+`, inline rename, close, restored badge (see §1). _Support:_ `CompanionTerminals.tsx`.
- **SessionSidebar / NewSessionDialog** — The legacy "session" model UI: header with The Octo at 18px (static; replaced the 🐙 emoji — the empty state's 4xl 🐙 is likewise now a 40px idle Octo) + "Octopush", a session list with status dot, icon, name, project•model, cost•tokens, budget gauge, tags; new-session dialog with template/name/root/model (an Atelier `Listbox`, "Default (agent decides)" as a real option; sets `OCTOPUSH_MODEL`)/icon/color/tags. _Support:_ `SessionSidebar.tsx`, `NewSessionDialog.tsx` (predates the Atelier redesign). _Entry:_ ⌘T.

### Notable implementation details
- **Why out-of-process** — PTYs are owned by their spawner; the split lets shells survive app restarts/auto-updates (like JetBrains/VS Code). Protocol v2 forces a one-time daemon replacement that deploys the eager-fd-release + `remove` fixes; running a pre-v2 and v2 build simultaneously makes them fight over the single daemon.
- **Restart flow** — App quits → daemon + shells keep running. Relaunch → reconnect, `list_terminals` reconciles DB rows with live daemon ids (`running`/`restored`); each `TerminalPane` calls `spawn_or_attach` (replays scrollback, fires `pty://reattached`).
- **Env injection summary** — Curated host-env allowlist + `TERM`/`COLORTERM` (daemon) → `HISTFILE`/`OCTOPUSH_PROJECT_TYPE`/`OCTOPUSH_GIT_BRANCH` (ContextGuard) → `OCTOPUSH_MODEL` (session create). Login shell in the project cwd.

## 8. Providers, Models, Tokens, Budgets & Usage

### Provider abstraction & wire protocols
- **`LlmProvider` trait** — One normalized `complete(api_base, api_key, req, client) -> LlmResponse` so `chat_engine`/`ai_complete`/orchestrator stay provider-agnostic. _Support:_ `providers/mod.rs` (`LlmRequest`, `LlmMessage`, `LlmContent`, `LlmBlock` {Text, Image}, `LlmTool`, `LlmResponse`, `LlmStopReason`).
- **Anthropic protocol adapter** — POSTs `{base}/v1/messages` with `x-api-key`, `anthropic-version`, output-128k beta; maps tools to `input_schema`, images as base64; parses `usage` incl. cache-read/creation. Requires a key. _Support:_ `providers/anthropic.rs`.
- **OpenAI-compatible adapter** — POSTs `{base}/chat/completions` with optional `Authorization: Bearer` (skipped when key empty, for Ollama); tools as functions, images as data URLs; parses `tool_calls`, `finish_reason`, `usage`. Covers OpenAI, DeepSeek, Ollama, vllm, llama.cpp, LMStudio, LocalAI. _Support:_ `providers/openai_compat.rs`.
- **Rate-limit headroom parsing** — Anthropic remaining-input-tokens + reset → `RateLimitSnapshot` so the agentic loop paces itself. _Support:_ `anthropic.rs::parse_rate_limit`.
- **Transient-failure retry** — `complete_with_retry` retries only transient errors (429/529/5xx/dropped), honoring `retry-after`; capped exponential backoff (max 60s, `DEFAULT_MAX_RETRIES = 5`), interruptible by a director stop. _Support:_ `providers/mod.rs`.

### Provider & model catalog (`provider_router.rs`)
- **`ProviderConfig`** — `name`, `api_base`, `api_key_env`, `models`, `rate_limits`, `enabled`, `protocol` (anthropic | openai-compatible), `local` (true ⇒ UI hides key, base URL editable). Persisted to `~/.octopush/providers.json`. _Support:_ `provider_router.rs`.
- **`ModelInfo` (per model)** — `id`, `display_name`, `input_cost_per_m`, `output_cost_per_m`, `cache_read_cost_per_m`, `cache_creation_cost_per_m`, `max_context`, `supports_vision`, `supports_tools`, `tags` (curated pills). _Support:_ `provider_router.rs`.

- **Catalog load + migrations** — `ProviderRouter::load()` seeds defaults on first run and applies on-disk migrations (protocol coercion, re-enable, backfill cache costs/tags/models, append missing builtins); reloaded fresh on every request (edits apply to the next message, no restart). _Support:_ `provider_router.rs::load`.
- **`list_models` (enabled-only, cost-sorted)** / **`find_model`** / **`suggest_model(TaskType)`** (8 task types → opus/sonnet/haiku with reason + cost tier) / **`validate_providers`**. _Support:_ commands `list_models`, `suggest_model`, `save_providers`, `get_default_providers`.
- **API keys & base-URL overrides stored separately** — keys/overrides live in `~/.octopush/settings.json` (`providerKeys`, `providerBaseUrls`), NOT in providers.json.

### Provider/Model management UI (Settings → Models)
- **Master-detail editor** — Provider list + detail pane; an "Unsaved changes" bar appears when the working copy diverges; Save validates server-side then read-modify-writes settings.json (merging keys/baseUrls so it doesn't wipe other settings) and re-fetches models. _Support:_ `settings/ModelsPane.tsx`.

- **Remove provider/model confirmation** — Routed through `ConfirmDialog`; removing a provider prunes its keys/baseUrls. _Support:_ `ModelsPane.tsx`.
- **Pricing refresh footer** — "Pricing · {relative time}" + spin-on-fetch refresh; refuses while there are unsaved edits; "Updated X of Y". _Support:_ `ModelsPane.tsx`.

### Pricing engine
- **In-process pricing table** — `cost_per_token(model, TokenType)` for built-in ids incl. cache rates; unknown/local = $0. _Support:_ `token_engine.rs`.
- **Cost computation (two paths)** — `compute_cost` (hardcoded table) and `compute_cost_with_prices` (explicit `ModelInfo` prices; non-Anthropic cache billed at input rate); `record` prefers the catalog-price path. _Support:_ `token_engine.rs`.

### CLI agent adapters (`agent_adapter.rs`)
- **`AgentAdapter` trait** — `name`, `display_name`, `build_command(model, cwd)`, `parse_token_usage`, `supports_hot_swap`. _Support:_ `agent_adapter.rs`.

### Token engine
- **`TokenEvent` schema** — `{session_id, timestamp, input/output/cache_read/cache_creation_tokens, model, cost_usd}` in the `token_events` table; `session_id` doubles as the workspace id. _Support:_ `token_engine.rs`, `db.rs`.
- **`TokenEngine::record`** — Computes cost if 0 (catalog-aware), inserts the event, increments the session aggregate. _Support:_ `token_engine.rs`.
- **Two recording modes** — **API usage** (authoritative): chat engine + `ai_complete` record real counts. **PTY scraping** (best-effort): `scan_pty_output` matches an embedded `usage` JSON block or a Claude-Code "Total cost: …" summary line. _Support:_ `chat_engine.rs`, `commands.rs`, `token_engine.rs`.
- **Canonical spend ledger (`spend_events`, Phase 2)** — A dedicated ledger table with an explicit `surface` (`talk`/`run`/`direct`/`review`/`adhoc`), denormalized `project_id`/`workspace_id` (project attribution survives a workspace deletion), `provider_cost_usd`/`computed_cost_usd` + `cost_basis`, and an `idempotency_key` (dedupe via `INSERT OR IGNORE`). **DIRECT run spend is now recorded here** — `complete_run_stage` mirrors each stage completion into the ledger (one row per completion, so every re-run/looped attempt stays counted), fixing the gap where orchestrator spend was invisible to Usage and budgets. A one-shot migration backfills historical DIRECT runs from `run_stages`. ### Notable implementation details
- **Two protocols, not N vendors** — Adding any vendor/gateway means choosing `anthropic` or `openai-compatible`; no per-vendor code.

- **Token-event schema** — `token_events` (TALK/RUN/REVIEW; `session_id` doubles as the workspace id) plus the canonical `spend_events` ledger (DIRECT, with explicit surface + attribution + idempotency); the `all_spend` view unions both, and project-scope budgets/savings join `all_spend → workspaces`.

- **PTY token scraping is best-effort and fragile by design** — the authoritative path is API usage recorded by `chat_engine`/`ai_complete`.

### The `octopush-mcp` standalone server binary
- **What/transport/scope** — A sidecar `[[bin]]` exposing a **read-and-author** slice of Octopush over JSON-RPC 2.0 stdio, reusing `octopush_lib::db` against the same SQLite store (WAL-safe while the app is open). **Never executes runs or spends tokens**; the one git mutation it performs is materialising a workspace's worktree (`create_workspace`). _Support:_ `bin/octopush-mcp/main.rs`; doc `docs/octopush-mcp.md`.
- **Protocol handshake** — Negotiates version (latest `2025-06-18`; also `2025-03-26`, `2024-11-05`), advertises `tools`, returns `instructions`; tool errors in-band (`isError: true`). _Support:_ `protocol.rs`.
- **Exposed tools (22)** — _reference:_ `describe_pipeline_schema` (now also documents the `customRoles` authoring contract). _read:_ `list_pipelines`, `get_pipeline`, `list_roles`, `list_projects`, `list_workspaces`, `get_workspace`, `list_runs`, `get_run`, `list_routines`. _author:_ `create_pipeline` (validated DAG), `update_pipeline` (forks a builtin / edits a custom), `delete_pipeline` (builtins protected), `save_role`/`delete_role` (see role authoring below), `create_routine`/`update_routine`/`delete_routine`/`set_routine_enabled` (see routine authoring below), `link_workspace_issue` (metadata-only), `create_run` (stages a DIRECT run in **`draft`** — not started), `create_workspace` (ensures a worktree-backed workspace via the shared `workspace::create`; an explicit branch is used **verbatim** (validated git ref), task-derived branches are slugified; adopts an already-checked-out branch rather than failing; returns `status: created|adopted|existed|restored`; appears in the rail on next focus/refresh). `create_pipeline`/`update_pipeline` author the **same per-stage knobs the app builder does**, including per-stage reasoning **`effort`** and the model-escalation policy (**`escalateModel`**/**`escalateEffort`**) — see the authoring contract below. The DB handle is a `Mutex<Db>` so `create_workspace` locks per-op, never across its git checkout. _Support:_ `tools.rs`.
- **Role authoring over MCP (`save_role`/`delete_role`)** — an agent can author the **custom roles a good pipeline depends on**, not just reference the built-ins. `save_role` creates/updates a custom role from just `label` + `promptBody` (every other field defaults to the app's new-role values: `artifactKind` note, `environment` worktree, `defaultTools` [read_file, list_files], `defaultSubstrate` api, `canLoop`/`defaultCheckpoint` false, `tokenEstIn`/`tokenEstOut` 4000/1000); the `key` (a stage's `role`) is derived from the label with the **exact** app algorithm (`deriveRoleKey`) when omitted, and always returned. Routes through the same `db.upsert_role`/`get_role`/`role_in_use`/`delete_role` the app command uses — **built-in keys are protected** (a save that would overwrite one, or a delete of one, is refused), out-of-enum `artifactKind`/`environment` is a **clean tool error** (typed `RoleDef` deserialize) and an unknown `defaultTools` entry or `defaultSubstrate` (stored on `RoleDef` as a plain `Vec<String>`/`String`, not enums) is rejected by an explicit check matching the advertised enums, and a role **still used by any pipeline/run stage** cannot be deleted. Like `delete_pipeline`, an MCP save/delete does **not** push to the Pro cloud library from the MCP process (edit once in-app to sync). _Support:_ `tools.rs` (`save_role`/`delete_role`/`list_roles`/`derive_role_key`), `db::upsert_role`/`role_in_use`. Design record: `docs/superpowers/plans/2026-07-15-mcp-authoring-parity-design.md`.
- **Routine authoring over MCP (`create_routine`/`update_routine`/`delete_routine`/`set_routine_enabled`)** — an agent can author **scheduled crews** (Pro routines), the full in-app Routines surface minus immediate firing. **Author-only, safe by default:** `create_routine` creates the routine **DISABLED** unless `enabled:true` is passed — authoring never silently schedules token spend; the user enables it in-app or the agent enables it on explicit instruction (`set_routine_enabled`). There is deliberately **no `run_routine_now`** (firing a crew *now* is execution, which stays in the app, like launching a run). An enabled routine fires only while the app runs and the plan includes `routines.scheduled` (**re-checked at fire time**, so an MCP-authored routine can't fire for a non-entitled user). `create_routine`/`update_routine` share one payload deserialized into the typed `RoutineInput` and validated through the **same** `octopush_lib::routines::{validate_routine, validate_schedule, next_due}` the app command uses, plus **referential-integrity checks** the agent needs (project/pipeline exist; `fixed` mode requires a `fixedWorkspaceId` that exists and belongs to the project; `fresh`⇒`daily`) so a dangling routine is rejected up front, not left to fail at fire time. `scheduleKind` `interval` (seconds-as-string ≥ 60) or `daily` (`HH:MM`); `stageModelOverrides` (`[position, model]` pairs) round-trip through the stored JSON string; the optional `fireCondition` (exit-0 pre-fire gate) is carried through. `update_routine` leaves `enabled` untouched; `delete_routine` is ungated (matches the app). _Support:_ `tools.rs` (`create_routine`/`update_routine`/`delete_routine`/`set_routine_enabled`/`list_routines`/`build_routine_input`), `db::insert_routine`/`update_routine`/`set_routine_enabled`, `routines::{validate_routine,validate_schedule,next_due}`. Design record: `docs/superpowers/plans/2026-07-15-mcp-authoring-parity-design.md`.
- **Pipeline authoring contract** — Same `§3.7` validator as the app's `save_pipeline` (roles, loops on review roles, acyclic parents, substrate, tools subset, maxIterations 1..100, instructions ≤8000). Per-stage inputs include the **cost/quality knobs**: `effort` (`low|medium|high|xhigh|max`; null/omit = off; API-only; takes effect only on reasoning-capable models — current Claude families + budget-path Haiku 4.5/Sonnet 4.5, where higher levels auto-clamp to the model's max; on legacy/unknown/non-Claude ids it is **silently ignored**, not clamped) and the escalation policy `escalateModel` (retry once on stage failure **only if it raises the tier** above `agentModel`; api+cli) / `escalateEffort` (api-only; retries only when higher than the base effort). Both round-trip through the exact `StageDraft`→`save_pipeline` path the app uses; `save_pipeline` normalizes a whitespace-only `escalateModel` to none, and an out-of-enum `effort`/`escalateEffort` is a clean per-stage tool error (never a panic). `describe_pipeline_schema` documents these fields with their real semantics, the effort/escalation **rules**, a `runtimeBehaviors` section (the `ask_director` **escape valve** — an **API-substrate** stage may pause to ask the director when blocked; a `cli` stage never self-blocks, use a checkpoint/escalation there; plus **auto-escalation** guidance), and an annotated example that sets `effort` + `escalateModel`. _Support:_ `tools.rs`.
- **Registration & namespacing** — `claude mcp add octopush -- <binary>`; tools appear as `mcp__octopush__<tool>`. Roadmap (not yet): execution control-plane, TALK/REVIEW surfaces, issue-tracker reads. _Support:_ `docs/octopush-mcp.md`.

### Skills
- **Discovery** — `<name>/SKILL.md` from `<worktree>/.claude/skills/*` (project) and `~/.claude/skills/*` (user); project shadows user. _Support:_ `skills/mod.rs`.
- **Frontmatter parsing** — Hand-rolled (no YAML dep): requires a `---` fence; fields `name` (required), `description`, `allowed-tools`/`tools`. _Support:_ `skills/mod.rs::parse_skill`.

### Notable implementation details
- **Where config lives** — Jira creds in `settings.json`; per-project Jira key + per-workspace `linked_issue_key` in SQLite; MCP-client servers in `~/.claude/mcp.json` (user) + `<repo>/.claude/mcp.json` (project); skills in `.claude/skills/*`; Claude-Code registration of octopush-mcp lives in Claude Code's own user-scope config.
- **MCP registry/transport** — Two directions: **client** (Octopush spawns external stdio servers, lazy-cached, namespaced `mcp__server__tool`, routed in the chat loop) and **server** (Octopush *is* a stdio MCP server for terminal CLIs). Both hand-rolled JSON-RPC 2.0 over newline-delimited stdio.
- **octopush-mcp safety scope** — communicated three ways (server `instructions`, each tool's description, the doc): never executes, never spends tokens, never mutates git (the lone git touch is `create_workspace`'s worktree); runs land in `draft` for the user to launch from the app's DIRECT mode; **routines are authored DISABLED by default and there is no `run_routine_now`** (the scheduler — in the app, gated per fire — is the only thing that ever fires one); drives the same DB whether or not the app is running.

commands `auth_begin_sign_in`/`auth_cancel_sign_in`/`auth_status`/`auth_refresh`/`auth_sync_plan`/`auth_sign_out`/`auth_account_portal_url`/`billing_checkout_url`; `lib/awaitPro.ts`. _Entry:_ Settings → Account (App group). _Mechanism:_ the OAuth 2.0 Authorization Code + **PKCE (S256)** **public-client** flow runs in the Rust core (`src-tauri/src/auth.rs`) — opens the system browser to Clerk, captures the redirect on a `127.0.0.1:8976` loopback (cancellable mid-flight) that serves a **branded Atelier confirmation page** (onyx/brass, distinct success + error states, dynamic text HTML-escaped), exchanges the code (**no client secret**), fetches `/oauth/userinfo`, and stores the session in the macOS **Keychain** (`keyring`). `auth_status` **silently refreshes** the access token when expired (signs out only on a revoked token, never on a transient/offline failure). **The plan rides on the OAuth session:** sign-in requests the Clerk **`public_metadata`** scope and reads `public_metadata.plan` (e.g. "pro") from userinfo into the session; `entitlement::current()` maps it to the entitlement (Pro = uncapped + all features). `auth_refresh` re-fetches userinfo to pick up a plan change. **After checkout the app auto-detects the upgrade** (`awaitPro.ts`): a bounded ~2-min lightweight poll (`auth_refresh`, no token rotation) **plus** a single forced token refresh (`auth_sync_plan`) when the window regains focus — a freshly-minted access token carries the latest `public_metadata`, so the badge + gates flip to **Pro on their own** (with a success toast), no manual sign-out/in. **"Upgrade"** (`billing.rs`) opens a Dodo **static checkout link** for the **live** $20/mo Pro product, stamped with the user's email + `metadata_clerk_user_id` (so a server-side webhook can map the subscription back **by Clerk id, not email**) — the desktop holds **no payment secret**. The plan is set server-side by a **Dodo→Clerk webhook** (a Vercel Function in the separate `octopush-api` repo) on `subscription.active`/`cancelled` (the webhook verifies against both the live and test signing secrets), and the **Free Direct-run cap (25/mo) is live** for non-Pro. **Dodo runs in live mode** (checkout host + product id baked into `billing.rs`), and **Clerk runs on the production instance** (custom domain `clerk.octopush.sh`; the public-client OAuth `client_id` + instance are baked into `auth.rs`, and the Account Portal resolves to `accounts.octopush.sh`).
- **Shared pane primitives** — `PaneHeader`, `SectionLabel`, `ToggleRow`, `Stat`, `Row`, `useChartColors` (live CSS-var theme colors for Recharts). _Support:_ `settings/shared.tsx`.

## Appendix D — Processes & on-disk locations

- **Processes** — the Tauri app (`octopush`), the PTY daemon (`octopush-pty-server`, out-of-process, launchd-adopted), and on demand the MCP server (`octopush-mcp`, stdio, spawned by a CLI like Claude Code). Direct CLI stages spawn `claude -p …`; Run terminals spawn the user's login shell.
- **`~/.octopush/`** — `settings.json` (keys, base URLs, git credentials, issue-tracker config, editor command, last pricing refresh), `providers.json` (provider/model catalog incl. pricing), `theme.json`, `projects/` (default scaffold location), `pty-server.sock` / `pty-server.pid` / `pty-server.log`, `pty-state/<id>.log` (terminal scrollback), `history/<session>.hist`.
- **`~/Library/Application Support/octopush/`** — `octopush.db` (SQLite: projects, workspaces, sessions, terminals, chats, pipelines, runs, roles, journals, token events, budgets, …).
- **`~/.claude/`** — read for `mcp.json` (MCP servers) and `skills/*` (user skills); Claude Code's own config receives the `octopush` MCP registration.
- **Per-project** — `.octopus-worktrees/<branch>/` (workspace worktrees), `.claude/skills/*` and `.claude/mcp.json` (project-scoped skills/servers), and the migration from legacy `~/.octopus-sh` / `octopus-sh` dirs runs once on launch.
- **Privacy posture** — local-first, no analytics/telemetry; outbound traffic only to configured AI providers and to integrations the user enables (Jira over HTTPS, GitHub via `gh`/REST).
